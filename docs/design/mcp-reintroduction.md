# MCP reintroduction on calfkit 0.9.0

Status: in progress (June 2026).

calfcord removed its MCP support in `9f81d3f` because calfkit's original MCP
adaptor was incorrectly designed: it required *manual schema registration*
(committed `schemas/<server>.py` modules generated by `calfkit mcp codegen`
against a live server) and shipped tool schemas as static agent-side imports.
calfkit 0.9.0 replaces that with a correct implementation — an `MCPToolbox`
node with **runtime tool discovery** over a compacted capability topic. This
doc records the design for reintroducing MCP on that API, plus the new
end-user onboarding surface (`mcp.json` + `calfcord mcp` CLI).

`docs/design/mcp-removal.md` enumerates everything the removal touched; this
design is its inverse where semantics were good (selector grammar, secrets
boundary, roster wiring) and a redesign where the old shape existed only to
serve codegen (catalog/discovery/schema_build are **not** coming back).

## What calfkit 0.9.0 provides

- `calfkit.mcp.mcp_toolbox.MCPToolbox(server_name, connection_params)` — a
  `BaseNodeDef` (`node_kind="toolbox"`) that owns the live MCP session. Hosted
  on a `Worker`, it connects at startup, `list_tools()`, and publishes a
  `CapabilityRecord` (tool names + JSON schemas + its dispatch topic
  `mcp_server.<name>`) to the compacted `mcp.capabilities` topic, re-publishing
  on `tools/list_changed` and on a heartbeat. Clean shutdown tombstones the
  record; a crash leaves it (agents keep last-known tools).
- `calfkit.mcp.mcp_transport.StdioServerParameters` (command/args/env/cwd) and
  `StreamableHttpParameters` (url/headers/timeouts).
- Agent-side: `Agent(tools=[...])` accepts `ToolSelector` objects — a
  runtime-checkable protocol (`resolve_tools(view) -> SelectorResult`)
  resolved **per turn** against a `KafkaTable[CapabilityRecord]` capability
  view that `Worker` auto-registers whenever a hosted node declares selectors.
  `calfkit.models.capability.resolve_capability(view, toolbox_id, include=,
  strict=)` is the public resolution helper.
- Non-strict selection degrades with a warning when the toolbox/tool is
  missing from the view; `strict=True` raises `MCPToolResolutionError`.

Key property: a selector never touches the MCP session or its config — it is
a pure capability-view lookup. That is what lets calfcord keep its old
**secrets boundary**: agent hosts need only the selector strings from the
agent `.md`; `mcp.json` (commands, URLs, credentials) is read exclusively by
the MCP server processes.

## Decisions

### D1. Per-server roster processes (not a singleton `mcp` component)

Each configured server runs as its own Process Compose slot `mcp-<server>`,
derived from `mcp.json` exactly like agent slots are derived from `agents/`.

- `MCPToolbox` fails its hosting `Worker` at boot when the server is
  unreachable. MCP server configs are user-supplied commands/URLs — the
  highest-misconfiguration-rate config in the system. One typo must not take
  down every MCP tool (the old singleton `calfkit-mcp` had exactly that flaw).
- Editing one entry only needs that one server restarted.
- The lifecycle surface mirrors agents, which operators already know:
  `calfcord mcp start|stop|restart <server>|--all`, with the roster's
  "start = restart in place" semantics. **`calfcord mcp start --all` is the
  "re-pick up mcp.json" command** — it regenerates compose state from the
  file and (re)starts every configured server.
- stdio servers are per-server subprocesses anyway; the singleton saved
  nothing real.

`mcp-<server>` process names are reserved alongside the existing reserved set;
the compose generator rejects an agent whose name collides with an MCP slot
(same fail-loud rule as the existing reserved-name guard).

### D2. `mcp.json` — Cursor/Claude Code schema, literals allowed

Location: `$CALFCORD_HOME/config/mcp.json` (next to `config/.env`), override
via `CALFCORD_MCP_CONFIG`, dev fallback `./mcp.json`. Seeded by the installer
as `{"mcpServers": {}}` with mode 0600, never clobbered on update.

Schema (same shape Cursor / Claude Code use, and same as the old calfcord
mcp.json):

```json
{
  "mcpServers": {
    "github": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-github"],
      "env": {"GITHUB_TOKEN": "$GITHUB_TOKEN"}
    },
    "docs": {
      "type": "http",
      "url": "https://docs.example.com/mcp",
      "headers": {"Authorization": "Bearer $DOCS_TOKEN"}
    }
  }
}
```

- stdio entries: `command` (string), optional `args` (list), optional `env`
  (map). HTTP entries: `"type": "http"`, `url`, optional `headers`.
- `$VAR` / `${VAR}` references are expanded against the process environment at
  load time (`$$` escapes a literal dollar). Unset references fail the load
  with the variable named. **Literal values are allowed** (matching the tools
  users copy configs from); `config/` already holds secrets in `.env` and the
  file is 0600. The wizard nudges secrets toward `$VAR` but does not block.
- Server names must match `[a-z0-9_]{1,64}` (doubles as the topic segment in
  `mcp_server.<name>` and the compose slot suffix).
- Loading is calfcord's own ~hundred lines (`calfcord.mcp.config`); calfkit
  0.9.0 deliberately has no config-file story.

### D3. Agent `.md` semantics — identical to pre-removal

- `mcp/<server>` → every tool the server advertises.
- `mcp/<server>/<tool>` → exactly that tool. Tool segment `[a-zA-Z0-9_-]{1,128}`.
- MCP tools are never part of the "tools key omitted → all builtins" default;
  they are always explicit.
- Parse-time (`definition._validate_tools`) and write-time
  (`md_writer.update_tools`) validate selector **syntax only**; whether the
  server is configured/up is a runtime concern (runtime discovery means there
  is no static catalog to check against — by design).

`agents/_mcp_guard.py` (the temporary rejection shim) is deleted; its
replacement is `calfcord.mcp.selector` restoring the old leaf grammar module
(`is_mcp_selector` / `parse_mcp_selector` / `McpSelector` NamedTuple /
`is_valid_server_name`).

### D4. Agent-side resolution — calfcord-owned selector, no `mcp.json` on agent hosts

`AgentFactory._resolve_tools` partitions `tools:` into builtin names (resolved
against `TOOL_REGISTRY`, unchanged) and MCP selectors, which are grouped per
server and mapped to one `McpToolSelector` per server:

```python
@dataclass(frozen=True)
class McpToolSelector:          # implements calfkit's ToolSelector protocol
    server: str
    include: tuple[str, ...] | None   # None = the bare mcp/<server> wildcard
    def resolve_tools(self, view): return resolve_capability(view, self.server, include=self.include)
```

- `mcp/<server>` plus `mcp/<server>/<tool>` for the same server merge the way
  the old schema_build did: a bare selector subsumes the explicit ones
  (include=None); explicit-only selections produce a deduplicated, sorted
  include tuple.
- Non-strict always (calfcord policy): an agent boots and runs even when its
  MCP servers are down; the turn logs the degradation. Strictness can be
  exposed later if a need appears.
- calfkit's own `MCPToolbox.select()` returns a private `_ScopedSelector` and
  requires constructing the toolbox (connection params) first — unusable on a
  distributed agent host. calfcord's selector is ~10 lines against the public
  `ToolSelector` protocol + `resolve_capability`. A calfkit issue will request
  a public by-name selector constructor so this can collapse later.
- The LLM-facing tool name is whatever the server advertises (no rename layer:
  the old `<server>_<tool>` rename existed for codegen-era collision rules).
  Collisions: calfkit drops a toolbox tool whose name collides with a static
  tool (static wins, logged) — factory additionally fails loud only for
  builtin-vs-builtin collisions as today.
- `phonebook.py` restores `_builtin_tools_only`: MCP selectors are not A2A
  capabilities and stay out of phonebook entries.
- The agent runner's `Worker` is built with default `MCPDiscoveryConfig`
  (capability topic `mcp.capabilities`); the view is auto-registered by
  calfkit only when an agent actually declares selectors.
- An import-isolation test再 guards the boundary: the agent path must never
  import `calfcord.mcp.config`.

### D5. Runner — `calfkit-mcp <server>`

Console script `calfkit-mcp` (module `calfcord.mcp.runner`): load `mcp.json`,
build the **one** named server's `MCPToolbox`, `Client.connect(...,
reply_topic="calfkit.mcp.reply", provisioning=PROVISIONING)`, single-node
`Worker`, `run_worker_until_signal`. Unknown/missing server name → actionable
`SystemExit` listing configured names. Compose slots run `calfcord run mcp
<server>` (shim → `calfkit-mcp <server>`).

### D6. CLI surface

- `calfcord mcp add` — **interactive wizard** (InquirerPy via the existing
  `Prompter` seam) when invoked bare: name → transport (stdio/HTTP) →
  command or URL → env/header loop → preview merged JSON → write → offer to
  start now. Flag-driven when any config flag is present (`--command`,
  `--url`, `--env`, `--header`, `--force`, `--dry-run`, `--start`) for
  scripting parity with the old `calfcord-mcp-add`.
- `calfcord mcp list` — configured servers (name, transport, summary) plus
  running state from the supervisor when available.
- `calfcord mcp remove <server>` — delete the entry (confirm; offers stop
  first if running).
- `calfcord mcp start|stop|restart <server>|--all` — roster lifecycle,
  mirroring `agent` verbs.
- `calfcord agent tools` / create-wizard checkbox: one `mcp/<server>` row per
  configured server; when the broker is reachable, per-tool
  `mcp/<server>/<tool>` rows are read live from the `mcp.capabilities`
  compacted topic (short catch-up timeout), degrading silently to server-level
  rows offline. Selectors present in the agent `.md` but not enumerable are
  preserved as pre-checked rows (never silently dropped — old behavior).
- `mcp codegen` does **not** return (nothing to generate), and `calfcord init`
  stays MCP-free (onboarding stays minimal; docs point to `calfcord mcp add`).

### D7. calfkit bump 0.7.0 → 0.9.0

Mechanical migration shipped in the same PR, first commit(s):

- `ConsumerNodeDef` → `ConsumerNode`; consume functions take
  `ConsumerContext` (a superset of the old `NodeResult` fields calfcord read:
  `output`, `state`, `correlation_id`, `emitter_node_id/kind`, `deps`);
  `output_type=` → `agent_output_type=`. Sites: `bridge/steps.py`,
  `bridge/outbox.py`, `bridge/synthesized.py`, `router/fanout.py`,
  `control_plane/first_reply.py`, `control_plane/state_consumer.py`.
- `Client.invoke_node(output_type=...)` and `ToolNodeDef`/`tool_schema`
  surfaces are unchanged; `0.8.0` removed `input_args`, which calfcord never
  used.
- New transitive deps: `mcp`, `ktables`.

## Topology after this change

Five process types again: bridge, agents, router, tools, **mcp servers** (one
process per configured server). The capability topic `mcp.capabilities` is a
compacted control-plane topic created by ktables' `ensure_topic` under
calfcord's enabled provisioning (production deploys create it out-of-band,
`cleanup.policy=compact`).
