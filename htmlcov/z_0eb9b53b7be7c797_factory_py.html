1"""Construct a runnable calfkit :class:`Worker` from an :class:`AgentDefinition`. 

2 

3The factory builds a vanilla :class:`calfkit.Agent` node — no subclassing — 

4configured to subscribe to: 

5 

6* a per-agent **private return topic** ``{agent_id}.private.return`` at 

7 index ``[0]`` of ``subscribe_topics`` (so it is also the callback topic 

8 for tool ``Call`` and ``TailCall`` envelopes — see below); 

9* ``discord.channel.{cid}.in`` for each channel in the agent's persisted 

10 state; 

11* a single per-agent inbox topic ``agent.{agent_id}.in`` used by the 

12 ``calfkit-tools`` runner to invoke the agent for A2A traffic without 

13 round-tripping through Discord. 

14 

15The agent's identity rides on every outbound publish via calfkit's 

16``x-calf-emitter`` Kafka header, so the bridge egress can resolve the 

17responding agent's persona from ``ConsumerContext.emitter_node_id`` without 

18any application-level identity stamping. 

19 

20**Why the private return topic at index [0]:** calfkit's 

21:class:`~calfkit.nodes.base.BaseNodeDef` treats 

22``subscribe_topics[0]`` as the callback topic for tool ``Call`` 

23envelopes and as the target for ``TailCall`` retries. When two 

24agents co-tenant on the same channel topic (the multi-agent ambient 

25fan-out this project relies on), making that shared channel the 

26callback would deliver each agent's tool return to *every* 

27co-tenant, causing each peer to also run its LLM on the originating 

28agent's state and emit a duplicate (and often incorrect) reply to 

29the user. Putting a per-agent private topic at ``[0]`` keeps tool 

30returns scoped to the agent that initiated the call; the channel 

31topics still live at ``[1:]`` so ambient fan-out is unaffected. The 

32name matches calfkit's own :attr:`BaseNodeDef._return_topic` 

33attribute — which calfkit ≥ 0.9.0 wires into the publish path and 

34auto-subscribes — so the explicit placement here is belt-and- 

35suspenders agreement with upstream, no longer a workaround. 

36 

37Tools declared in the agent's ``.md`` frontmatter under ``tools:`` are 

38resolved against :data:`calfcord.tools.TOOL_REGISTRY` and 

39passed to the calfkit ``Agent`` constructor. Each agent only carries the 

40tool's :class:`~calfkit.nodes.ToolNodeDef` for schema + subscribe-topic 

41purposes — the actual tool body runs in the ``calfkit-tools`` deployment. 

42 

43Two public entry points: 

44 

45* :meth:`AgentFactory.build_node` returns a bare :class:`Agent`. The 

46 ``calfkit-agent`` runner uses this in both single-agent and all-agents 

47 modes — the runner constructs the :class:`Worker` itself so it can pack 

48 one or many nodes into a single Worker depending on invocation. 

49* :meth:`AgentFactory.build` is a thin convenience that wraps 

50 :meth:`build_node` in a one-node :class:`Worker`. Kept for callers that 

51 want a complete Worker without assembling it themselves (and for the 

52 existing ``test_factory.py`` suite); the in-tree runner does not use 

53 it. 

54 

55The factory dispatches on :attr:`AgentDefinition.provider` to choose between 

56:class:`calfkit.AnthropicModelClient` and :class:`calfkit.OpenAIModelClient`. 

57Each provider has a default model name baked in (see 

58:data:`_PROVIDER_DEFAULT_MODELS`); a definition's ``model`` field wins, the 

59``CALFKIT_AGENT_DEFAULT_MODEL`` env var is the next fallback, and the 

60provider-specific default is the last resort. 

61 

62The factory does NOT manage the lifecycle of its dependencies 

63(``DiscordPersonaSender`` and calfkit :class:`Client`) — they are owned by 

64:mod:`calfcord.agents.runner` and passed in already-started. 

65 

66**Worker subscription is fixed at boot.** Calfkit's 

67:meth:`Worker.register_handlers` is one-shot, so mutating 

68``state.channels`` at runtime (via ``store.add_channel``/``store.remove_channel``) 

69does not change the running agent's Kafka subscriptions. Adding a channel 

70to an existing agent requires a process restart. The ``store`` parameter 

71is accepted into :meth:`build` for forward compatibility but is not used 

72in v1 — ``thinking_effort`` (the other runtime-tunable knob) lives in the 

73``.md`` frontmatter rather than the state file, so the store is purely a 

74channels reader. 

75"""

229class AgentFactory: 

230 """Builds a :class:`Worker` running one LLM-backed agent. 

231 

232 Identity carriage: 

233 The constructed :class:`Agent` is unmodified calfkit; calfkit 

234 stamps ``x-calf-emitter`` / ``x-calf-emitter-kind`` Kafka headers on 

235 every outbound publish, so the bridge resolves the responding agent 

236 via :attr:`~calfkit.models.ConsumerContext.emitter_node_id` and looks up the persona in 

237 the :class:`AgentRegistry`. 

238 

239 Provider selection priority (first non-empty wins): 

240 1. ``definition.provider`` 

241 2. ``os.environ["CALFKIT_AGENT_DEFAULT_PROVIDER"]`` 

242 3. ``default_provider`` constructor argument (defaults to 

243 ``"anthropic"``) 

244 

245 Model selection priority (first non-empty wins): 

246 1. ``definition.model`` 

247 2. ``os.environ["CALFKIT_AGENT_DEFAULT_MODEL"]`` 

248 3. ``default_model`` constructor argument (if set) 

249 4. The resolved provider's entry in 

250 :data:`_PROVIDER_DEFAULT_MODELS` 

251 """

330 def build_node( 

331 self, 

332 definition: AgentDefinition, 

333 state: AgentRuntimeState | None, 

334 store: AgentStateStore | None, 

335 ) -> Agent: 

336 """Build one :class:`Agent` node from ``definition`` and ``state``. 

337 

338 Use this when assembling several agents into a single :class:`Worker` 

339 (multi-node co-tenancy). Each node gets its own Kafka consumer group 

340 keyed on its ``node_id``, so co-tenant nodes do not contend for 

341 partitions. 

342 

343 Routers (``definition.role == "router"``) take a separate build 

344 path: single-topic subscription on the configured ambient topic, 

345 no standard gates, ``ToolOutput`` final-output type, explicit 

346 ``publish_topic``. See :meth:`_build_router_node`. Routers 

347 accept ``state=None``/``store=None`` because their subscription 

348 list does not depend on per-channel state. 

349 

350 Raises: 

351 ValueError: If ``definition.role == "assistant"`` and 

352 ``state`` is ``None`` or ``state.channels`` is empty 

353 (assistants must subscribe to at least one channel), 

354 or if the resolved provider isn't one of 

355 :data:`_PROVIDER_DEFAULT_MODELS` (typically only 

356 reachable via env-var override with an unknown 

357 value), or if ``definition.memory`` is set but the 

358 agent lacks the ``read_file``/``write_file`` tools 

359 memory requires (see :meth:`_require_memory_tools`), 

360 or if a router definition violates router-specific 

361 invariants. 

362 """ 

363 if definition.role == "router": 

364 return self._build_router_node(definition) 

365 

366 if state is None or not state.channels: 

367 raise ValueError( 

368 f"agent {definition.agent_id!r} has no channels in state; " 

369 "an assistant must subscribe to at least one channel" 

370 ) 

371 tools, mcp_selectors = self._resolve_tools(definition) 

372 self._require_memory_tools(definition, tools) 

373 

374 provider = self._resolve_provider(definition) 

375 model_name = self._resolve_model(definition, provider) 

376 # Per-agent private return topic at index [0]: calfkit uses 

377 # ``subscribe_topics[0]`` as the callback for tool ``Call`` 

378 # envelopes and the ``TailCall`` retry target. Channel topics 

379 # are shared across co-tenant agents (the ambient fan-out 

380 # pattern), so they must NOT be index 0 or each agent would 

381 # receive the others' tool returns and emit duplicate replies. 

382 # See the module docstring for the full failure mode. 

383 subscribe_topics = [_PRIVATE_RETURN_TOPIC_TEMPLATE.format(agent_id=definition.agent_id)] 

384 subscribe_topics.extend(self._subscribe_topic_template.format(cid=cid) for cid in state.channels) 

385 # Per-agent inbox: the ``calfkit-tools`` runner publishes A2A 

386 # invocations to this topic so the LLM-driven ``private_chat`` 

387 # tool can reach this agent without round-tripping through Discord. 

388 subscribe_topics.append(_AGENT_INBOX_TOPIC_TEMPLATE.format(agent_id=definition.agent_id)) 

389 model_settings = build_model_settings(provider, definition.thinking_effort) 

390 

391 logger.info( 

392 "building agent=%s provider=%s model=%s topics=%s thinking_effort=%s tools=%s", 

393 definition.agent_id, 

394 provider, 

395 # ``None`` for an unconfigured Codex model — resolved to the live 

396 # catalog default at client construction (logged there). 

397 model_name if model_name is not None else "<codex catalog default>", 

398 subscribe_topics, 

399 definition.thinking_effort, 

400 [t.tool_schema.name for t in tools] 

401 + [f"mcp:{s.toolbox_id}" for s in mcp_selectors], 

402 ) 

403 

404 # ``publish_topic=AGENT_STEPS_TOPIC`` makes FastStream mirror every 

405 # handler hop (``Call`` / ``TailCall`` / ``ReturnCall``) to a shared 

406 # audit feed the bridge's steps consumer subscribes to. The action- 

407 # specific publish (``ReturnCall`` → frame.callback_topic, ``Call`` → 

408 # tool topic) still happens through ``BaseNodeDef._publish_action``; 

409 # the publish_topic is an additional mirror, not a replacement. 

410 # The frontmatter-side ``publish_topic`` field stays assistant- 

411 # prohibited (``AgentDefinition._validate_router_constraints``) 

412 # because the injection here is a factory-level concern, not an 

413 # operator-tunable. See ``topics.AGENT_STEPS_TOPIC`` for the 

414 # single-partition operator requirement on this topic. 

415 agent = Agent( 

416 node_id=definition.agent_id, 

417 system_prompt=definition.system_prompt, 

418 subscribe_topics=subscribe_topics, 

419 publish_topic=AGENT_STEPS_TOPIC, 

420 model_client=self._model_client_factory(provider, model_name), 

421 model_settings=model_settings, 

422 tools=[*tools, *mcp_selectors] or None, 

423 ) 

424 agent.gate(make_addressable_gate(definition.agent_id)) 

425 agent.gate(make_addressed_to_me_gate(definition.agent_id)) 

426 

427 # Memory-enabled agents carry a dynamic-instructions hook. It reads 

428 # the bridge-injected template from ``deps`` at runtime, localizes it 

429 # to this agent's ``memory/<agent_id>/`` dir, and appends it to the 

430 # instructions. The agent process never reads the prompt file — only 

431 # the bridge does (see ``agents/memory.py``). ``_require_memory_tools`` 

432 # above already guaranteed the agent has the fs tools the block tells 

433 # it to use. 

434 if definition.memory: 

435 agent.instructions(memory_instructions(definition.agent_id)) 

436 

437 # ``store`` is accepted for forward compatibility but unused; 

438 # see module docstring on why runtime channel changes aren't 

439 # wired yet. We deliberately do not bind it locally. 

440 _ = store 

441 

442 return agent

444 def _build_router_node( 

445 self, 

446 definition: AgentDefinition, 

447 ) -> Agent: 

448 """Build the router :class:`Agent` node. 

449 

450 Router invariants enforced by :class:`AgentDefinition`: 

451 - ``tools`` is empty (the router uses :class:`ToolOutput`, 

452 not function tools) 

453 - ``publish_topic`` is non-None (declares the router's own 

454 output topic; the fan-out consumer subscribes there) 

455 

456 Differences from the assistant build path: 

457 - No state channels required (router subscribes to a single 

458 fixed ambient ingress topic, not per-channel topics). 

459 - No standard gates (no self-recognition, no 

460 addressed-to-me check — the router is the only consumer 

461 of its ingress topic, so every envelope is for it). 

462 - ``final_output_type=ToolOutput(RoutingDecision, 

463 name=ROUTER_OUTPUT_TOOL_NAME)`` so the LLM's structured 

464 output terminates the agent loop in one turn (pydantic-ai 

465 recognizes the tool call as the agent's terminal output 

466 under ``end_strategy="early"``). 

467 - ``publish_topic`` is set, so FastStream's ``@publisher`` 

468 wrapping mirrors the agent's ``ReturnCall`` to that topic. 

469 The duplicate publish to ``frame.callback_topic`` lands 

470 on a throwaway topic with no consumer (see 

471 ``bridge/ingress.py``). 

472 

473 Router builds need no per-channel state (single fixed ambient 

474 topic) and no on-disk state store. Callers should reach this 

475 method via :meth:`build_node` with ``state=None``/``store=None`` 

476 rather than building a never-used :class:`AgentStateStore`. 

477 

478 Note on provider precedence: the router definition's 

479 ``provider``/``model`` are populated by 

480 :func:`build_router_definition` from the bundled ``router.md`` 

481 front matter (with in-code defaults). Those values are always 

482 non-None, so the assistant-targeted ``CALFKIT_AGENT_DEFAULT_*`` 

483 env vars never apply on this path. A future refactor that lets 

484 the router definition leave ``provider`` or ``model`` as 

485 ``None`` would silently start picking up the assistant 

486 defaults — keep them non-None. 

487 """ 

488 provider = self._resolve_provider(definition) 

489 model_name = self._resolve_model(definition, provider) 

490 model_settings = build_model_settings(provider, definition.thinking_effort) 

491 

492 subscribe_topics = [AMBIENT_INGRESS_TOPIC] 

493 

494 logger.info( 

495 "building router agent=%s provider=%s model=%s topic=%s publish_topic=%s", 

496 definition.agent_id, 

497 provider, 

498 model_name, 

499 subscribe_topics[0], 

500 definition.publish_topic, 

501 ) 

502 

503 agent = Agent( 

504 node_id=definition.agent_id, 

505 system_prompt=definition.system_prompt, 

506 subscribe_topics=subscribe_topics, 

507 publish_topic=definition.publish_topic, 

508 model_client=self._model_client_factory(provider, model_name), 

509 model_settings=model_settings, 

510 final_output_type=ToolOutput(RoutingDecision, name=ROUTER_OUTPUT_TOOL_NAME), 

511 ) 

512 # No standard gates: the router is the only consumer of its 

513 # ingress topic, so every envelope is for it. 

514 # :meth:`BridgeIngress.handle`'s ambient branch already drops 

515 # ``is_bot`` / ``is_webhook`` authors before publishing 

516 # (preventing agent-on-agent reply storms), and a 

517 # "self-author" loop is impossible because the router never

519 return agent

540 def _resolve_tools( 

541 self, definition: AgentDefinition 

542 ) -> tuple[list[ToolNodeDef], list[MCPToolboxRef]]: 

543 """Resolve ``definition.tools`` into builtin nodes + deferred MCP selectors. 

544 

545 The flat ``tools:`` list mixes two kinds of entry, partitioned here: 

546 

547 * bare *builtin* names (``terminal``, ``calendar``) resolve against the 

548 in-memory tool registry, with an aggregate unknown-name 

549 :class:`ValueError` (every unknown name in one message); 

550 * ``mcp/...`` selectors collapse into one 

551 :class:`~calfkit.mcp.MCPToolboxRef` per server 

552 (:func:`~calfcord.mcp.agent_select.selectors_from_entries`), 

553 resolved per turn against the capability view — never against any 

554 local registry, so there is nothing further to validate here. 

555 

556 Name collisions between the two kinds cannot be checked statically 

557 (MCP tool names are only known at runtime); calfkit's per-turn 

558 resolution drops a toolbox tool that collides with a static binding 

559 (static wins, logged), which is the operative policy. 

560 

561 Semantics (mirrors :attr:`AgentDefinition.tools`): 

562 - ``None``: tools-by-default — every registered builtin tool and 

563 **no** MCP tools (MCP is always an explicit grant). This is 

564 the in-memory representation of the "no ``tools:`` line in 

565 frontmatter" case before the loader normalizes it. 

566 - empty tuple ``()``: zero tools (explicit opt-out). 

567 - non-empty tuple: exactly those entries. 

568 

569 Raises: 

570 ValueError: if any builtin name is missing from the registry 

571 (lists every unknown name in one message). 

572 """ 

573 if definition.tools is None: 

574 return list(self._tool_registry.values()), [] 

575 if not definition.tools: 

576 return [], [] 

577 

578 mcp_entries = [e for e in definition.tools if is_mcp_selector(e)] 

579 nodes: list[ToolNodeDef] = [] 

580 unknown: list[str] = [] 

581 for name in definition.tools: 

582 if is_mcp_selector(name): 

583 continue 

584 node = self._tool_registry.get(name) 

585 if node is None: 

586 unknown.append(name) 

587 else: 

588 nodes.append(node) 

589 if unknown: 

590 known = sorted(self._tool_registry) 

591 raise ValueError( 

592 f"agent {definition.agent_id!r} declares unknown tool(s) " 

593 f"{unknown!r}; known tools: {known or '<none registered>'}" 

594 ) 

595 

596 return nodes, selectors_from_entries(mcp_entries)

598 def _require_memory_tools(self, definition: AgentDefinition, tools: list[ToolNodeDef]) -> None: 

599 """Reject a ``memory: true`` agent that lacks the filesystem tools memory needs. 

600 

601 A memory-enabled agent manages its notepad with the general-purpose 

602 ``read_file`` / ``write_file`` tools; without them the injected memory 

603 instructions are a silent no-op, so fail loud at build time instead. 

604 Agents that omit ``tools:`` get every registered tool and pass 

605 automatically — only an explicitly-restricted ``tools:`` list can trip 

606 this. Operates on the resolved :class:`~calfkit.nodes.tool.ToolNodeDef` 

607 list (not the raw names) so the "all tools" expansion is reflected 

608 correctly. 

609 """ 

610 if not definition.memory: 

611 return 

612 available = {t.tool_schema.name for t in tools} 

613 missing = sorted({"read_file", "write_file"} - available) 

614 if missing: 

615 raise ValueError( 

616 f"agent {definition.agent_id!r} sets memory: true but is missing " 

617 f"required filesystem tool(s) {missing}; memory needs read_file and " 

618 f"write_file. Add them to the agent's tools:, or omit tools: to grant all." 

619 )
