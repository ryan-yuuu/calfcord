1"""Opt-in Kafka topic-provisioning policy for the runners. 

2 

3calfkit auto-provisions topics on broker start (opt-in via ``ProvisioningConfig`` 

4on ``Client.connect``), letting calfcord run on brokers that do not auto-create 

5topics — notably Tansu — for everything its startup ensurer can declare: 

6 

7* The **client reply topic** is auto-provisioned on EVERY ``broker.start()`` 

8 path (a connect-time pre-start hook; calf-ai/calfkit-sdk#180). calfcord no 

9 longer provisions it anywhere. 

10* A **Worker's node topics** are auto-provisioned on the *managed* run surfaces 

11 (``Worker.run()`` / ``start()`` / ``async with``), whose ``_on_startup`` hook 

12 declares :func:`~calfkit.provisioning.topics_for_nodes` into the same ensurer. 

13 The **tools and agents runners** use the embedded managed ``Worker.start()`` 

14 surface, so every Worker-hosted process gets its node topics for free. The 

15 **bridge is a pure** :class:`~calfkit.client.Client` (no Worker, no consumers) 

16 — it hosts no nodes and so declares no node topics; its only topic is the 

17 client reply topic, covered by the pre-start hook above. 

18 

19After the calfkit 0.12 migration removed the bespoke control plane, calfcord has 

20no blind-spot topics left to declare: agent presence and the live roster now ride 

21calfkit's native mesh (``calf.agents``), which the framework owns end-to-end, and 

22A2A is native tool/handoff dispatch over ordinary node topics the managed ensurer 

23already covers. So :data:`PROVISIONING` is the only piece every runner still 

24needs; :func:`provision_extra_topics` / :func:`provision_and_start_broker` remain 

25as general-purpose helpers for any future caller that hand-rolls a raw broker 

26subscriber outside the managed Worker lifecycle. 

27""" 

28 

29from __future__ import annotations 

30 

31from collections.abc import Iterable 

32from typing import TYPE_CHECKING 

33 

34from calfkit import ProvisioningConfig 

35from calfkit.provisioning import TopicProvisioner 

36 

37if TYPE_CHECKING: 

38 from calfkit.client import Client 

39 

40PROVISIONING = ProvisioningConfig(enabled=True, num_partitions=1, replication_factor=1) 

41"""Shared opt-in provisioning policy passed to every calfcord ``Client.connect``. 

42 

43``enabled`` so calfkit creates referenced topics on a broker without 

44auto-creation; ``num_partitions=1`` is the single-partition local/dev default 

45(nothing calfcord runs locally benefits from more, and a single partition keeps 

46per-key ordering trivially intact); ``replication_factor=1`` is the single-broker 

47local/dev default (NOT durable — raise it for a real multi-broker cluster)."""

50async def provision_extra_topics(server_urls: str | Iterable[str], topics: Iterable[str]) -> None: 

51 """Create ``topics`` that calfkit's startup ensurer cannot declare. 

52 

53 A no-op when ``topics`` is empty (no admin client is constructed). Otherwise 

54 de-duplicates (first-seen order) and creates them with :data:`PROVISIONING` 

55 via calfkit's :class:`~calfkit.provisioning.TopicProvisioner`, against 

56 ``server_urls`` — the same bootstrap the caller passes to ``Client.connect`` 

57 (calfkit 0.6.0 removed ``Client.server_urls``, so the URL(s) must be threaded 

58 through explicitly). calfcord configures no broker security, so no 

59 ``security_kwargs`` are forwarded; the standalone provisioner connects with a 

60 plaintext admin client like the rest of calfcord. Idempotent — already-existing 

61 topics are reported, not recreated. These are plain data topics, so 

62 ``framework_topics`` is empty. 

63 """ 

64 topic_list = list(dict.fromkeys(topics)) 

65 if not topic_list: 

66 return 

67 provisioner = TopicProvisioner.from_connection( 

68 server_urls=server_urls, 

69 config=PROVISIONING, 

70 ) 

71 report = await provisioner.provision(topic_list, framework_topics=set()) 

72 # calfkit's provisioner does NOT raise when the broker authorizes the 

73 # connection but DENIES create (ACL code 29): it records the topic in 

74 # ``report.unauthorized`` and only logs a warning. Swallowing that lets the 

75 # runner come up and then silently stall on the wire (a raw subscriber or a 

76 # publish to a topic that never gets created), so raise loudly instead — 

77 # calfcord's infra-failure-raises rule. Operators must pre-create these 

78 # out-of-band when the broker enforces CREATE ACLs. 

79 if report.unauthorized: 

80 raise RuntimeError( 

81 f"topic provisioning unauthorized for {sorted(report.unauthorized)} on " 

82 f"broker {server_urls}: the broker denies CREATE (ACLs). Pre-create these " 

83 f"out-of-band; a consumer/publisher would otherwise stall on the missing " 

84 f"topic. (created={sorted(report.created)}, existing={sorted(report.existing)})" 

85 )

88async def provision_and_start_broker( 

89 client: Client, 

90 server_urls: str | Iterable[str], 

91 topics: Iterable[str], 

92) -> None: 

93 """Provision ``topics`` the ensurer can't see, then bring the broker up. 

94 

95 For any caller that hand-rolls a raw ``broker.subscriber(...)`` followed by a 

96 bare ``broker.start()``, bypassing the managed Worker lifecycle — so calfkit's 

97 ``_on_startup`` ensurer never fires and does NOT provision its topics. On a 

98 broker that does not auto-create topics that bare start blocks forever unless 

99 every subscribed topic already exists, so the caller passes its blind-spot set 

100 here and they are created BEFORE the bare start. 

101 

102 The client reply topic is intentionally NOT included: calfkit's connect-hook 

103 auto-provisions it on every start path (calf-ai/calfkit-sdk#180), so the 

104 ``broker.start()`` below creates it for free. 

105 

106 Idempotent on an auto-creating broker (every create reports "existing"). The 

107 ``broker.running`` guard avoids a non-idempotent second ``start()`` if the 

108 broker was already brought up (e.g. by an earlier lazy first publish). 

109 

110 The production runners do NOT need this helper: the tools/agents runners use 

111 the managed ``Worker.start()`` surface (whose lifecycle auto-provisions reply 

112 + node topics), and the bridge is a pure :class:`~calfkit.client.Client` that 

113 hand-rolls no broker (its reply topic is covered by the connect-time pre-start 

114 hook). None of them declare blind-spot topics, so nothing here is needed — 

115 this stays a general-purpose helper for a future raw-broker caller. 

116 """ 

117 await provision_extra_topics(server_urls, topics) 

118 if not client.broker.running: 

119 await client.broker.start()
