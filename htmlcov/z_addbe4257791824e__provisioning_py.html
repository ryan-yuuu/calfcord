1"""Opt-in Kafka topic-provisioning policy and blind-spot helpers for the runners. 

2 

3calfkit 0.6.0 auto-provisions topics on broker start (opt-in via 

4``ProvisioningConfig`` on ``Client.connect``), letting calfcord run on brokers 

5that do not auto-create topics — notably Tansu — but only for what its startup 

6ensurer can declare: 

7 

8* The **client reply topic** is auto-provisioned on EVERY ``broker.start()`` 

9 path (a connect-time pre-start hook; calf-ai/calfkit-sdk#180). calfcord no 

10 longer provisions it anywhere. 

11* A **Worker's node topics** are auto-provisioned only on the *managed* run 

12 surfaces (``Worker.run()`` / ``start()`` / ``async with``), whose 

13 ``_on_startup`` hook declares :func:`~calfkit.provisioning.topics_for_nodes` 

14 into the same ensurer. The tools/router/agents runners use ``Worker.run()`` 

15 and the bridge uses the embedded ``Worker.start()`` — all managed surfaces — so 

16 every Worker-hosted process gets its node topics for free. 

17 

18Two gaps remain that calfcord fills explicitly via :func:`provision_extra_topics` 

19(and, before a bare start, :func:`provision_and_start_broker`): 

20 

21* **Hand-rolled node topics.** The control-plane probe deliberately decomposes 

22 ``Worker.run()`` — wiring a raw ``broker.subscriber(...)`` and then a bare 

23 ``broker.start()`` — because it owns a one-shot read (see 

24 ``docs/design/calfkit-worker-lifecycle-gaps.md``). The managed ``_on_startup`` 

25 ensurer never fires for it, so it provisions its (blind-spot-only) topics 

26 itself before the bare start. (The bridge USED to be such a caller; it now 

27 folds onto the embedded ``Worker.start()`` managed surface, so calfkit 

28 auto-provisions its node topics and it only declares its blind spots — below.) 

29* **Blind-spot topics** that ``topics_for_nodes()`` cannot see at all: raw 

30 FastStream broker subscribers and boot-time publish targets. The per-runner 

31 ``*_infra_topics`` sets below name exactly which. On the managed runners these 

32 are declared into the client's startup ensurer from a worker ``on_startup`` 

33 hook so they ride calfkit's single pre-start provisioning pass: the agents 

34 runner declares :func:`agent_infra_topics` and the bridge declares 

35 :func:`bridge_infra_topics` that way. (The router has no blind-spot topics: 

36 since ambient sends use ``reply_to=None``, there is no terminal-callback 

37 target to provision — its only topics are node ``subscribe_topics`` / 

38 ``publish_topic`` the managed ensurer already covers.) 

39 

40Why these particular extras (and not the rest of the cross-process contracts in 

41``calfcord.topics`` / ``calfcord.control_plane.topics``): every other shared 

42topic is also a node ``subscribe_topics``/``publish_topic`` on the process that 

43needs it created early, so the ensurer already covers it. Only the raw 

44control-plane subscribers and the boot-time publishes (which cannot wait for the 

45peer process to create the topic) fall through — those are listed here. 

46"""

65PROVISIONING = ProvisioningConfig(enabled=True, num_partitions=1, replication_factor=1) 

66"""Shared opt-in provisioning policy passed to every calfcord ``Client.connect``. 

67 

68``enabled`` so calfkit creates referenced topics on a broker without 

69auto-creation; ``num_partitions=1`` because ``agent.steps`` ordering REQUIRES a 

70single partition (see :data:`calfcord.topics.AGENT_STEPS_TOPIC`) and nothing 

71local benefits from more; ``replication_factor=1`` is the single-broker 

72local/dev default (NOT durable — raise it for a real multi-broker cluster)."""

74# The bridge<->agent control-plane pair both sides touch: the bridge subscribes 

75# agent.state and publishes bridge.discovery at boot; agents subscribe 

76# bridge.discovery and publish agent.state at boot. Shared so the two infra sets 

77# that include it cannot drift apart. 

78_SHARED_CONTROL_PLANE_TOPICS = (AGENT_STATE_TOPIC, BRIDGE_DISCOVERY_TOPIC)

81def bridge_infra_topics() -> list[str]: 

82 """Non-node topics the bridge must ensure exist before ``broker.start()``. 

83 

84 ``agent.state`` is consumed by a raw broker subscriber (the state consumer, 

85 not a Worker node). ``bridge.discovery`` is *published* at boot (the 

86 discovery ping) possibly before any agent is up, so the bridge cannot rely 

87 on an agent having created it. 

88 

89 Declared into the client's startup ensurer from the bridge's 

90 ``Worker.on_startup`` hook (pre-broker-start; 

91 :func:`calfcord.bridge.gateway._register_blind_spot_topics`), so they are 

92 created in calfkit's single managed provisioning pass alongside the node 

93 topics + reply topic, before the raw state consumer's group joins or the 

94 discovery ping publishes. 

95 """ 

96 return list(_SHARED_CONTROL_PLANE_TOPICS)

99def agent_infra_topics(agent_ids: Iterable[str]) -> list[str]: 

100 """Non-node topics the agents runner must ensure exist before ``broker.start()``. 

101 

102 The shared control-plane pair (``bridge.discovery`` subscribed by the raw 

103 control sink, ``agent.state`` published at boot before the bridge may be up) 

104 plus one ``agent.{id}.control.in`` per hosted agent (each a raw control-sink 

105 subscriber). 

106 

107 Declared into the client's startup ensurer from the agents runner's 

108 ``Worker.on_startup`` hook (pre-broker-start), so they are created in 

109 calfkit's single managed provisioning pass alongside the node topics + 

110 reply topic, before any raw control sink consumes or the presence publish 

111 fires. 

112 """ 

113 return [*_SHARED_CONTROL_PLANE_TOPICS, *(control_topic_for(agent_id) for agent_id in agent_ids)]

116async def provision_extra_topics(server_urls: str | Iterable[str], topics: Iterable[str]) -> None: 

117 """Create ``topics`` that calfkit's startup ensurer cannot declare. 

118 

119 A no-op when ``topics`` is empty (no admin client is constructed). Otherwise 

120 de-duplicates (first-seen order) and creates them with :data:`PROVISIONING` 

121 via calfkit's :class:`~calfkit.provisioning.TopicProvisioner`, against 

122 ``server_urls`` — the same bootstrap the caller passes to ``Client.connect`` 

123 (calfkit 0.6.0 removed ``Client.server_urls``, so the URL(s) must be threaded 

124 through explicitly). calfcord configures no broker security, so no 

125 ``security_kwargs`` are forwarded; the standalone provisioner connects with a 

126 plaintext admin client like the rest of calfcord. Idempotent — already-existing 

127 topics are reported, not recreated. These are plain data topics, so 

128 ``framework_topics`` is empty. 

129 """ 

130 topic_list = list(dict.fromkeys(topics)) 

131 if not topic_list: 

132 return 

133 provisioner = TopicProvisioner.from_connection( 

134 server_urls=server_urls, 

135 config=PROVISIONING, 

136 ) 

137 report = await provisioner.provision(topic_list, framework_topics=set()) 

138 # calfkit's provisioner does NOT raise when the broker authorizes the 

139 # connection but DENIES create (ACL code 29): it records the topic in 

140 # ``report.unauthorized`` and only logs a warning. Swallowing that lets the 

141 # runner come up and then silently stall on the wire (a raw subscriber or a 

142 # publish to a topic that never gets created), so raise loudly instead — 

143 # calfcord's infra-failure-raises rule. Operators must pre-create these 

144 # out-of-band when the broker enforces CREATE ACLs. 

145 if report.unauthorized: 

146 raise RuntimeError( 

147 f"topic provisioning unauthorized for {sorted(report.unauthorized)} on " 

148 f"broker {server_urls}: the broker denies CREATE (ACLs). Pre-create these " 

149 f"out-of-band; a consumer/publisher would otherwise stall on the missing " 

150 f"topic. (created={sorted(report.created)}, existing={sorted(report.existing)})" 

151 )

154async def provision_and_start_broker( 

155 client: Client, 

156 server_urls: str | Iterable[str], 

157 topics: Iterable[str], 

158) -> None: 

159 """Provision ``topics`` the ensurer can't see, then bring the broker up. 

160 

161 For the **hand-rolled** control-plane probe only: it wires a raw 

162 ``broker.subscriber(...)`` and then a bare ``broker.start()``, bypassing the 

163 managed Worker lifecycle — so calfkit's ``_on_startup`` ensurer never fires 

164 and does NOT provision its topics. On a broker that does not auto-create 

165 topics that bare start blocks forever unless every subscribed topic already 

166 exists, so the caller passes its blind-spot set here and they are created 

167 BEFORE the bare start. 

168 

169 The client reply topic is intentionally NOT included: calfkit 0.6.0's 

170 connect-hook auto-provisions it on every start path (calf-ai/calfkit-sdk#180), 

171 so the broker.start() below creates it for free. 

172 

173 Idempotent on an auto-creating broker (every create reports "existing"). The 

174 ``broker.running`` guard avoids a non-idempotent second ``start()`` if the 

175 broker was already brought up (e.g. by an earlier lazy first publish). 

176 

177 The Worker-hosted runners do NOT use this helper: tools/router/agents run 

178 via ``Worker.run()`` and the bridge via the embedded ``Worker.start()``, 

179 whose managed lifecycle auto-provisions reply + node topics; the router has 

180 no blind-spot topics at all, and the agents runner and bridge declare theirs 

181 into the startup ensurer from an ``on_startup`` hook — all letting the 

182 managed broker start do the provisioning. 

183 """ 

184 await provision_extra_topics(server_urls, topics) 

185 if not client.broker.running: 

186 await client.broker.start()
