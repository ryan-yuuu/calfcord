1"""Project-wide Kafka topic constants for cross-process contracts. 

2 

3Topics that are produced by one process and consumed by another live 

4here so producer and consumer always agree on the literal. Putting 

5these in a tiny zero-dependency module lets any package 

6(``bridge/``, ``router/``, ``agents/``) import them without risking 

7import cycles, and removes the need for "drift-guard" contract tests 

8that re-assert the same string in two places. 

9 

10Add a topic here only when **multiple processes** subscribe or publish 

11to it. Per-agent topics (``agent.{id}.in``, channel topics) stay where 

12they're consumed — they're parameterized strings, not cross-process 

13contracts. 

14""" 

15 

16from __future__ import annotations 

17 

18DISCORD_OUTBOX_TOPIC = "discord.outbox" 

19"""Topic every agent reply lands on; the bridge posts each one to Discord. 

20 

21An agent reply is the agent node's ``ReturnCall`` envelope: assistant agents 

22have ``publish_topic=None`` and emit to the inbound frame's ``callback_topic``, 

23which the bridge sets to this topic. The replying agent's identity is NOT in the 

24JSON body — it rides the Kafka headers (``x-calf-emitter`` / 

25``x-calf-emitter-kind``) and is recoverable only through a calfkit consumer 

26handler's ``_stamp_transport``. 

27 

28Producer: every assistant :class:`~calfkit.nodes.Agent` (via the bridge-set 

29``callback_topic``) — the bridge's ingress and retry sites publish with 

30``send(reply_to="discord.outbox")``, which sets each invocation's 

31``callback_topic`` to this topic. The bridge :class:`~calfkit.Client` does NOT 

32name it as its own ``reply_topic`` (it takes a private auto-generated inbox); the 

33topic is provisioned on broker start as the outbox consumer's 

34``subscribe_topics`` via the managed :class:`~calfkit.Worker` lifecycle (see 

35:mod:`calfcord.bridge.gateway`). 

36 

37Consumers: 

38* the bridge's outbox consumer 

39 (:func:`calfcord.bridge.outbox.build_outbox_consumer`), which posts each reply 

40 to Discord under the agent's persona; 

41* the init wizard's first-reply detector 

42 (:func:`calfcord.control_plane.first_reply.wait_for_first_reply`), a transient 

43 CLI-side consumer in its own group that confirms the org is live.""" 

44 

45AMBIENT_INGRESS_TOPIC = "discord.ambient.in" 

46"""Topic the router agent subscribes to. The bridge publishes ambient 

47(``kind="message"``) wires here for the router to classify. 

48 

49Producer: :mod:`calfcord.bridge.ingress` (ambient branch). 

50Consumer: the built-in router agent 

51(:mod:`calfcord.router.runner`, via the ``subscribe_topics`` 

52list built in :func:`calfcord.agents.factory.AgentFactory._build_router_node`). 

53 

54Hardcoded rather than env-driven because the project's topology contract 

55is fixed — operators changing this topic would also need to retune the 

56router's subscribe_topics, which is also constant."""

65Producer: :mod:`calfcord.router.fanout`.

76AGENT_STEPS_TOPIC = "agent.steps" 

77"""Shared topic that every assistant agent mirrors its handler hops to. 

78 

79FastStream's ``@publisher`` decorator (see 

80:meth:`calfkit.worker.Worker.register_handlers`) re-publishes every 

81``Response`` returned from a node's handler onto its ``publish_topic``. 

82Wiring this topic as the assistant agent's ``publish_topic`` makes 

83every intermediate ``Call`` / ``TailCall`` / ``ReturnCall`` envelope 

84visible on a single feed. 

85 

86Producer: every assistant :class:`~calfkit.nodes.Agent` built by 

87:meth:`calfcord.agents.factory.AgentFactory.build_node`. 

88Consumer: the bridge's steps consumer 

89(:func:`calfcord.bridge.steps.build_steps_consumer`), 

90which projects the message-history delta on each hop into a Discord 

91transcript thread. 

92 

93**Operator note — single partition required.** This topic MUST be 

94configured with a single partition (or all of one agent's hops must 

95hash to the same partition by some other means). FastStream's 

96``@publisher`` decorator wraps the calfkit handler's plain 

97``faststream.Response`` return without carrying a Kafka key, so on a 

98multi-partition topic the hops for one ``correlation_id`` can 

99round-robin partitions and arrive at the consumer out of order. The 

100consumer's history-cursor advance is monotonic, so an out-of-order 

101late hop would silently swallow its delta; an intermediate hop 

102arriving after a terminal hop would seed a second (unlocked) 

103transcript thread. The bridge's direct 

104:meth:`~calfkit.Client.publish` calls do stamp the correlation_id as 

105the partition key (see ``calfkit/nodes/base.py``); the gap is only 

106the publisher-decorator mirror path that ``publish_topic`` activates. 

107 

108Recommended Kafka config: 

109 

110 kafka-topics.sh --create --topic agent.steps --partitions 1 ..."""
