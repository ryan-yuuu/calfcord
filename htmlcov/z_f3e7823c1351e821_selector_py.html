1"""Parsing + validation for ``mcp/...`` tool selectors in agent frontmatter. 

2 

3Background 

4---------- 

5 

6Agents declare their tools in ``agents/*.md`` frontmatter under a single 

7``tools:`` list. Builtin tools appear there by bare name (``terminal``, 

8``web_search``); MCP-server tools appear in the *same* list using a ``mcp/`` 

9selector syntax so an author never has to learn a second declaration 

10mechanism: 

11 

12* ``mcp/<server>`` — expose *every* tool the named MCP server publishes. 

13* ``mcp/<server>/<tool>`` — expose a single tool of that server. 

14 

15The agent's LLM sees each tool under the name the MCP server advertises; 

16tool calls dispatch to the server's calfkit toolbox over the 

17``mcp_server.<server>`` topic. Resolving a selector against the live 

18capability view is handled downstream in :mod:`calfcord.mcp.agent_select`; 

19this module is concerned only with recognizing and decomposing the 

20selector string itself. 

21 

22Design choices 

23-------------- 

24 

25* **Leaf module, zero project imports** — this file imports nothing from 

26 :mod:`calfcord` or :mod:`calfkit` (only :mod:`re` + stdlib). The agent 

27 frontmatter parser (``calfcord.agents.definition``) needs to recognize 

28 and validate selectors *before* anything decides whether to build the 

29 schema-only MCP catalog. Keeping this module a pure leaf lets that 

30 parser ``from calfcord.mcp.selector import ...`` without dragging in 

31 the catalog build (which imports :mod:`calfkit`) or risking an import 

32 cycle with the ``tools`` package. 

33 

34* **Regexes redeclared, not imported** — the character-class rules below 

35 intentionally duplicate the spirit of ``TOOL_NAME_REGEX`` in 

36 :mod:`calfcord.tools.deploy_filters` rather than importing it. Importing 

37 from the ``tools`` package would create a coupling (and a potential 

38 import cycle, since tool modules import back through bridge/agent code) 

39 for the sake of one shared constant. A few lines of duplication here 

40 buys this module its leaf status; the regexes are commented so a future 

41 reader knows the omission is deliberate. 

42 

43* **Two distinct name grammars** — the *server* segment is constrained to 

44 ``[a-z0-9_]`` because it must double as a Kafka topic segment 

45 (``mcp_server.<server>``), an ``mcp.json`` key, and the suffix of the 

46 server's roster process name (``mcp-<server>``); lowercase-plus- 

47 underscore is the safe intersection. 

48 The *tool* segment allows ``[a-zA-Z0-9_-]`` (and a longer bound) to 

49 match the original MCP tool name as advertised by the upstream server, 

50 which we do not control and which commonly uses mixed case or hyphens. 

51 

52* **Strict, message-rich ``ValueError``** — every rejection names the 

53 offending ``entry`` verbatim so a typo in an ``agents/*.md`` file 

54 surfaces with the exact bad string, not a generic "invalid selector". 

55""" 

56 

57from __future__ import annotations

59import re 

60from typing import NamedTuple

62MCP_SELECTOR_PREFIX = "mcp/" 

63"""The literal prefix that marks a ``tools:`` entry as an MCP selector 

64rather than a builtin tool name. A bare entry like ``terminal`` is a builtin; 

65``mcp/gmail`` or ``mcp/gmail/search`` is an MCP selector.""" 

66 

67# Server segment: must double as a Kafka topic segment, an mcp.json key, 

68# and a roster process-name suffix, so we restrict to lowercase + digits + 

69# underscore. Redeclared here (rather than imported from 

70# calfcord.tools.deploy_filters) to keep this module a pure leaf — see the 

71# module docstring's "Regexes redeclared" note. 

72_SERVER_NAME_REGEX = re.compile(r"^[a-z0-9_]{1,64}$") 

73 

74# Tool segment: matches the *original* MCP tool name advertised by an 

75# upstream server we do not control; allow mixed case + hyphen + a longer 

76# bound, mirroring ``TOOL_NAME_REGEX`` in calfcord.tools.deploy_filters (also 

77# redeclared, not imported — same leaf-module rationale). 

78_TOOL_NAME_REGEX = re.compile(r"^[a-zA-Z0-9_-]{1,128}$")

81def is_valid_server_name(name: str) -> bool: 

82 """Return whether ``name`` is a legal ``<server>`` selector segment. 

83 

84 Single source of truth for "is this a valid MCP server name?" — it 

85 matches against the *same* module-level :data:`_SERVER_NAME_REGEX` 

86 (``^[a-z0-9_]{1,64}$``) that :func:`parse_mcp_selector` enforces on the 

87 server segment, so a name accepted here is exactly a name a ``mcp/...`` 

88 selector can reach. Reuses the compiled regex rather than re-spelling 

89 the pattern, so the grammar cannot drift between the two call sites. 

90 

91 The intended consumers are the ``mcp.json`` loader and the 

92 ``disco mcp add`` writer, which key server config by this name: a 

93 configured server whose name is not a legal server segment would be 

94 unreachable by any valid ``mcp/...`` selector (and could not name a 

95 Kafka topic segment or roster process). Validating here lets both 

96 reject that case loudly instead. 

97 

98 Args: 

99 name: A candidate server name (typically a ``schemas/`` module 

100 name). Not a full ``mcp/...`` selector — just the bare server 

101 segment. 

102 

103 Returns: 

104 ``True`` if ``name`` matches the server-segment grammar, else 

105 ``False``. 

106 """ 

107 return bool(_SERVER_NAME_REGEX.match(name))

110def is_mcp_selector(entry: str) -> bool: 

111 """Return ``True`` if ``entry`` is an MCP selector (vs. a builtin name). 

112 

113 A cheap prefix check only — it does *not* validate the rest of the 

114 selector. Callers that need a structurally-valid selector should run 

115 :func:`validate_mcp_selector` (or :func:`parse_mcp_selector`) after 

116 this returns ``True``. Splitting "is it ours?" from "is it valid?" 

117 lets the frontmatter parser route a ``mcp/`` entry into MCP handling 

118 and *then* report a precise parse error, rather than silently treating 

119 a malformed ``mcp/`` entry as an (always-unknown) builtin name. 

120 """ 

121 return entry.startswith(MCP_SELECTOR_PREFIX)

124class McpSelector(NamedTuple): 

125 """A parsed ``mcp/...`` selector: a server plus an optional single tool. 

126 

127 ``tool is None`` means "all tools of ``server``" (the bare ``mcp/<server>`` 

128 form); a concrete ``tool`` selects exactly one (``mcp/<server>/<tool>``). 

129 Prefer reading that distinction through :attr:`selects_all_tools` over 

130 re-checking ``tool is None`` at each call site. 

131 

132 As a :class:`~typing.NamedTuple` it stays backward compatible with the 

133 previous ``(server, tool)`` return: positional unpacking 

134 (``server, tool = parse_mcp_selector(...)``) and tuple equality 

135 (``== ("gmail", "search")``) both still hold. 

136 """ 

137 

138 server: str 

139 tool: str | None 

140 

141 @property 

142 def selects_all_tools(self) -> bool: 

143 """Whether this selector expands to every tool of :attr:`server`.""" 

144 return self.tool is None

147def parse_mcp_selector(entry: str) -> McpSelector: 

148 """Decompose an MCP selector into ``(server, tool_or_none)``. 

149 

150 Examples:: 

151 

152 parse_mcp_selector("mcp/gmail") -> ("gmail", None) 

153 parse_mcp_selector("mcp/gmail/search") -> ("gmail", "search") 

154 

155 A ``None`` tool means "all tools of this server" (the bare-server 

156 form); a non-``None`` tool selects exactly one. 

157 

158 Args: 

159 entry: The raw ``tools:`` list entry. Must start with 

160 :data:`MCP_SELECTOR_PREFIX`. 

161 

162 Returns: 

163 An :class:`McpSelector` (a ``(server, tool)`` NamedTuple) where 

164 ``tool`` is ``None`` for the bare-server form and the original tool 

165 name otherwise. It unpacks and compares as the plain ``(server, 

166 tool)`` tuple it replaced, so existing callers are unaffected. 

167 

168 Raises: 

169 ValueError: When ``entry`` does not start with the ``mcp/`` 

170 prefix, splits into a segment count other than 2 or 3, has an 

171 empty server or tool segment, or has a server/tool segment 

172 that violates the respective name grammar. The message always 

173 names ``entry`` verbatim so the offending frontmatter line is 

174 unambiguous. 

175 """ 

176 if not entry.startswith(MCP_SELECTOR_PREFIX): 

177 raise ValueError( 

178 f"MCP selector {entry!r} must start with {MCP_SELECTOR_PREFIX!r}" 

179 ) 

180 

181 # Split the WHOLE entry (not the post-prefix remainder) so the segment 

182 # count is checked against the documented forms directly: 

183 # "mcp/gmail" -> ["mcp", "gmail"] (len 2) 

184 # "mcp/gmail/search" -> ["mcp", "gmail", "search"] (len 3) 

185 # Anything else (a trailing slash, an extra path segment, a doubled 

186 # slash producing an empty middle) lands outside {2, 3} or trips the 

187 # emptiness checks below. 

188 segments = entry.split("/") 

189 if len(segments) not in (2, 3): 

190 raise ValueError( 

191 f"MCP selector {entry!r} must be 'mcp/<server>' or " 

192 f"'mcp/<server>/<tool>', got {len(segments)} '/'-separated " 

193 f"segment(s)" 

194 ) 

195 

196 server = segments[1] 

197 tool = segments[2] if len(segments) == 3 else None 

198 

199 if not server: 

200 raise ValueError(f"MCP selector {entry!r} has an empty server segment") 

201 if not _SERVER_NAME_REGEX.match(server): 

202 raise ValueError( 

203 f"MCP selector {entry!r} has invalid server name {server!r}; " 

204 f"must match {_SERVER_NAME_REGEX.pattern}" 

205 ) 

206 

207 if tool is not None: 

208 if not tool: 

209 raise ValueError(f"MCP selector {entry!r} has an empty tool segment") 

210 if not _TOOL_NAME_REGEX.match(tool): 

211 raise ValueError( 

212 f"MCP selector {entry!r} has invalid tool name {tool!r}; " 

213 f"must match {_TOOL_NAME_REGEX.pattern}" 

214 ) 

215 

216 return McpSelector(server, tool)

221 

222 Thin wrapper over :func:`parse_mcp_selector` that discards the parsed 

223 result — for call sites (e.g. the frontmatter validator) that care 

224 only about *whether* a selector is structurally valid, not about its 

225 decomposed parts. Keeping this as a named function makes those call 

226 sites read as the assertion they are and avoids a stray 

227 ``parse_mcp_selector(entry)``-with-unused-result lint smell. 

228 """ 

229 parse_mcp_selector(entry)
