1"""Discord → calfkit publish path: fire-and-forget agent invocation. 

2 

3The bridge publishes the normalized :class:`WireMessage` to the right 

4ingress topic based on ``wire.kind`` and returns immediately. Replies 

5land on ``discord.outbox`` and are posted by the outbox consumer 

6(:mod:`calfcord.bridge.outbox`) — every reply, not just the 

7first to win the calfkit reply dispatcher's ``correlation_id`` race. 

8 

9Kind-branching: 

10 

11* ``kind="slash"`` (a Discord slash command, an @-mention parsed as a 

12 slash, OR a synthesized wire from the router's fan-out): published 

13 to ``discord.channel.{cid}.in`` where every assistant agent 

14 subscribed to the channel sees it. The agent's 

15 ``addressed_to_me_gate`` accepts iff ``slash_target`` matches its 

16 own ``agent_id``. 

17* ``kind="message"`` (ambient — non-slash, non-@-mention text from a 

18 human): published to the router's ambient ingress 

19 (``discord.ambient.in``) with the original wire, the phonebook, and 

20 the channel history on ``deps``. The router's fan-out consumer reads 

21 them back from ``result.deps`` (calfkit ≥ 0.4.0), decides which 

22 assistants should respond, and republishes synthesized ``kind="slash"`` 

23 wires through ``bridge.synthesized.in``, which loops back to this 

24 class's :meth:`handle` via the bridge's synthesized-in consumer. 

25 

26Two pieces of state cross the ingress→egress boundary: 

27 

28* ``deps={"discord": wire}`` rides on the envelope so the agent's gates 

29 (:mod:`calfcord.agents.gates`) can read it. The dep 

30 survives the agent's :class:`ReturnCall` republish into 

31 ``discord.outbox`` because :meth:`BaseNodeDef._publish_action` carries 

32 ``envelope.context.deps`` forward. 

33* The same wire is also written to a process-local :class:`PendingWires` 

34 map keyed on ``correlation_id``, together with the bridge-computed 

35 retry context (history snapshot + cursor, temp instructions, model 

36 settings) the outbox needs to rebuild a faithful retry envelope — 

37 context that does NOT ride on ``ConsumerContext.deps`` the way the 

38 wire itself does. See :mod:`pending_wires` for the rationale. 

39 

40Per-call thinking-effort overrides: when ``wire.slash_target`` is set 

41we read the target agent's current ``thinking_effort`` from the 

42in-memory registry and attach a provider-specific ``model_settings`` 

43dict to the invocation so the agent uses the configured effort on this 

44exact call. Ambient messages flow without an override (the agent falls 

45back to whatever was baked into its model client at boot). 

46 

47Ambient-reply health signal (v1): 

48 We log INFO at every ambient publish and (in 

49 :mod:`bridge.synthesized`) INFO at every synthesized-in arrival. 

50 Operators correlating those two streams can detect a silent 

51 router. Per-reply WARN tracking (pairing each publish with its 

52 eventual user-visible reply) is deferred — keeps v1 simple. See 

53 ``docs/ambient-routing.md`` "Operating" section for the runbook. 

54"""

102REPLAY_TOOL_RETURN_MAX_CHARS: Final[int] = 6000 

103"""Per-tool-return character cap applied to replayed (hydrated) tool 

104returns before they re-enter an agent's ``message_history``. 

105 

106Deliberately distinct from — and much larger than — the 2000-char Discord 

107display budget the steps renderer targets: the Discord cap is a *display* 

108budget, while this one bounds the *LLM context* a replayed turn re-injects. 

109Reusing a display-sized render cap would lobotomize the tool context the 

110model gets to reason over on the next turn (the whole point of replay). 

111Only oversized individual ``str`` 

112tool returns are trimmed; ``history_turns`` already bounds how many turns 

113replay can touch (plan §4: "``history_turns`` bounds replay; no backstop" 

114and §11 Q-5: this is the lever that keeps the hydrated envelope under the 

115broker's max message size).""" 

116 

117_REPLAY_TRUNCATION_MARKER: Final[str] = "\n…(truncated)" 

118"""Visible marker appended to a tool return truncated to 

119:data:`REPLAY_TOOL_RETURN_MAX_CHARS` so the model can tell the content was 

120cut rather than genuinely short."""

123def _truncate_replay_tool_returns(messages: list[ModelMessage]) -> list[ModelMessage]: 

124 """Cap oversized tool-return payloads in a replayed delta. 

125 

126 Walks ``messages`` (the deserialized structured slice of a prior turn) 

127 and, for every :class:`BaseToolReturnPart` (so both plain 

128 :class:`ToolReturnPart` *and* :class:`BuiltinToolReturnPart`) whose 

129 model-facing string — ``part.model_response_str()`` — exceeds 

130 :data:`REPLAY_TOOL_RETURN_MAX_CHARS`, replaces the part with an 

131 immutable copy whose ``content`` is the truncated string (plus 

132 :data:`_REPLAY_TRUNCATION_MARKER`). This catches the cases the old 

133 ``str``-only check missed: a non-``str`` ``content`` (a structured 

134 payload) that serializes large, and ``BuiltinToolReturnPart``. Swapping 

135 a structured ``content`` for a truncated ``str`` is acceptable because 

136 the model only ever sees ``model_response_str()`` regardless — and a 

137 ``str`` round-trips through that method unchanged. 

138 

139 Already-small returns are left untouched; messages with no oversized 

140 tool returns are returned unchanged (the same object), and a message 

141 that does need trimming is rebuilt with :func:`dataclasses.replace` on 

142 the affected parts so the deserialized objects are never mutated in 

143 place. 

144 

145 Returns a NEW list — callers can splice it into the hydration map 

146 without worrying about aliasing the input. The non-mutating contract 

147 matters because the deserialized parts may be shared once the 

148 type-adapter caches anything, and because :func:`project_history` 

149 treats its hydration deltas as read-only. 

150 """ 

151 out: list[ModelMessage] = [] 

152 for msg in messages: 

153 new_parts: list[Any] | None = None 

154 parts = msg.parts 

155 for idx, part in enumerate(parts): 

156 if isinstance(part, BaseToolReturnPart): 

157 rendered = part.model_response_str() 

158 if len(rendered) > REPLAY_TOOL_RETURN_MAX_CHARS: 

159 if new_parts is None: 159 ↛ 161line 159 didn't jump to line 161 because the condition on line 159 was always true

160 new_parts = list(parts) 

161 head = REPLAY_TOOL_RETURN_MAX_CHARS - len(_REPLAY_TRUNCATION_MARKER) 

162 truncated = rendered[: max(head, 0)] + _REPLAY_TRUNCATION_MARKER 

163 new_parts[idx] = dataclasses.replace(part, content=truncated) 

164 if new_parts is None: 

165 out.append(msg) 

166 else: 

167 out.append(dataclasses.replace(msg, parts=new_parts)) 

168 return out

171class AmbientRosterEmptyError(ValueError): 

172 """Raised when an ambient publish is aborted because the registry 

173 has no eligible assistant agents. 

174 

175 The router LLM run would be useless (it would receive no roster, 

176 likely hallucinate, and the fan-out's phonebook validation would 

177 reject every chosen id anyway), and the user would see no reply. 

178 The gateway catches this specific exception and sends an 

179 operator-actionable inline reply to the Discord message — making 

180 the misconfiguration visible to the user instead of silently 

181 dropping the message. 

182 

183 Carries the original Discord ``event_id`` and ``channel_id`` for 

184 the gateway's reply text and any structured-logging downstream. 

185 """

197class BridgeIngress: 

198 """Publish inbound Discord events as agent invocations. 

199 

200 Fire-and-forget. The eventual reply (or replies — multi-agent flows 

201 are the point of the migration) is posted by the outbox consumer, 

202 not by this class. The slash command path and the ambient-message 

203 path share this single handler. 

204 """ 

205 

206 def __init__( 

207 self, 

208 calfkit_client: Client, 

209 registry: AgentRegistry, 

210 pending_wires: PendingWires, 

211 *, 

212 default_provider: Provider = DEFAULT_PROVIDER, 

213 ingress_topic_template: str = _DEFAULT_INGRESS_TOPIC_TEMPLATE, 

214 ) -> None: 

215 self._client = calfkit_client 

216 self._registry = registry 

217 self._pending_wires = pending_wires 

218 self._default_provider = default_provider 

219 self._ingress_topic_template = ingress_topic_template 

220 # The history fetcher is injected via :meth:`set_fetcher` after 

221 # the gateway client connects (its WebSocket-paired 

222 # :class:`discord.Client` isn't usable until ``_on_ready`` fires). 

223 # ``None`` is the documented pre-ready state — :meth:`handle` 

224 # degrades gracefully to empty history when the fetcher isn't 

225 # set yet so a Discord event arriving in the brief window 

226 # before ``_on_ready`` doesn't crash the invocation path. 

227 self._fetcher: ChannelHistoryFetcher | None = None 

228 # The transcript store is injected via :meth:`set_transcript_store` 

229 # once the bridge's SQLite connection is open (gateway ``main()``, 

230 # right after the ``async with TranscriptStore(...)`` enters). 

231 # ``None`` is the documented pre-ready state — the slash history 

232 # builder degrades to today's no-replay projection when the store 

233 # isn't set yet, so a Discord event arriving before the store opens 

234 # never crashes the invocation path. Mirrors :attr:`_fetcher`. 

235 self._transcript_store: TranscriptStoreLike | None = None 

236 # Whether the memory-prompt load has already failed once this process — 

237 # gates the one-shot error log in :meth:`_memory_prompt_deps` (re-armed 

238 # on a later successful load). Same late-bound-state idiom as the two 

239 # fields above. 

240 self._memory_prompt_load_failed = False 

241 # Validate every agent's provider at boot so a typo'd 

242 # CALFKIT_AGENT_DEFAULT_PROVIDER surfaces here (fail-fast) rather 

243 # than as an uncaught ValueError inside every targeted invocation. 

244 for spec in registry.all(): 

245 resolve_provider(spec, default_provider=default_provider) 

246 # Symmetrically validate that every `tools:` reference in every 

247 # .md resolves against TOOL_REGISTRY. The agent runner runs the same 

248 # check at its own boot, but the bridge usually starts first in dev — 

249 # surfacing typos here gives operators a single actionable error before 

250 # any agent process boots. (``mcp/...`` selectors are rejected far 

251 # earlier, at parse time in ``AgentDefinition._validate_tools``, so a 

252 # stray one would already be an unknown-tool reference here.) 

253 # Lazy import: `calfcord.tools` transitively imports bridge code, so a 

254 # top-level import would cycle at boot. 

255 from calfcord.tools import TOOL_REGISTRY 

256 

257 unknown: list[tuple[str, str]] = [] 

258 for spec in registry.all(): 

259 # ``spec.tools is None`` means "all registered tools" — the 

260 # loader's default-resolution sentinel for assistants whose 

261 # frontmatter omits the ``tools:`` line. Nothing to validate 

262 # (every name comes from TOOL_REGISTRY by construction). 

263 if spec.tools is None: 

264 continue 

265 for tool_name in spec.tools: 

266 if tool_name not in TOOL_REGISTRY: 

267 unknown.append((spec.agent_id, tool_name)) 

268 if unknown: 

269 known = sorted(TOOL_REGISTRY) 

270 entries = ", ".join(f"{aid!r} declares {tname!r}" for aid, tname in unknown) 

271 raise ValueError( 

272 f"bridge boot found unknown tool references: {entries}; " 

273 f"known tools: {known or '<none registered>'}"

306 async def handle( 

307 self, 

308 wire: WireMessage, 

309 *, 

310 prefetched_history: Sequence[HistoryRecord] | None = None, 

311 ) -> None: 

312 """Publish ``wire`` to the right ingress topic for its kind. 

313 

314 Slash wires (``kind="slash"``) — including real Discord 

315 slashes, parsed @-mentions, and the router's synthesized 

316 fan-out wires — go to the per-channel ingress topic. Ambient 

317 wires (``kind="message"``) go to the router's ambient ingress 

318 with the wire packed onto ``deps`` so the router's 

319 fan-out consumer can recover it from ``result.deps``. 

320 

321 ``prefetched_history`` is populated only by the synthesized-in 

322 consumer (:mod:`calfcord.bridge.synthesized`) which 

323 reads the history off ``deps["history"]`` and forwards it 

324 here. This avoids a redundant per-target Discord fetch when one 

325 ambient publish fans out to multiple agents — the bridge made 

326 a single fetch at ambient publish time, packed it onto 

327 ``deps``, and the synthesized-in consumer hands the same raw 

328 records to each target's invocation here. The per-agent POV 

329 projection happens locally in this method. 

330 

331 Only the slash branch writes to :class:`PendingWires` (before 

332 publishing, so a fast agent reply can never race the 

333 outbox's lookup). The ambient branch deliberately does NOT 

334 populate ``PendingWires``: the router's reply lands on the 

335 discard topic and is never looked up by the original 

336 ``event_id``; the synthesized wires the fan-out generates 

337 each carry their own fresh ``event_id``, and those are what 

338 the outbox correlates on. See the inline comment at the 

339 ambient branch for the LRU-eviction failure mode that 

340 skipping prevents. 

341 

342 Per-call ``model_settings`` resolution: when 

343 ``wire.slash_target`` is set, we look up the target agent's 

344 current ``thinking_effort`` in the registry and build a 

345 provider-specific override. Ambient messages send no override 

346 (the router's effort comes from its own definition). 

347 

348 Cancelling the invocation handle's future immediately after 

349 publish is load-bearing on both branches: :meth:`Client.invoke_node` 

350 unconditionally registers a pending future with the reply 

351 dispatcher (the dispatcher will otherwise resolve+pop it on 

352 the first reply, but a no-reply event would leak ``_pending`` 

353 forever, and a redelivered ``correlation_id`` would raise 

354 inside ``_ReplyDispatcher.expect``). Cancelling the future 

355 triggers its ``add_done_callback`` to pop the registry entry 

356 synchronously. The bridge's egress is the outbox consumer in 

357 a different consumer group, so the actual reply is still 

358 observed. 

359 """ 

360 # Build the phonebook fresh on every invocation so any future 

361 # hot-add on the registry takes effect immediately. The same 

362 # phonebook is used twice: locally to compute temp_instructions, 

363 # and serialized into deps so decoupled deployments (e.g. the 

364 # tools runner) can do persona lookups and peer-roster building 

365 # without needing local file access to agents/*.md. 

366 phonebook = phonebook_from_registry(self._registry) 

367 

368 if wire.kind == "message": 

369 # Ambient path. Filter non-human authors before publish: 

370 # peer-agent webhook chatter without an @-mention must not 

371 # trigger the router (would risk agent-on-agent reply 

372 # storms). Agent @-mentions still route normally because 

373 # the normalizer classifies on content, producing 

374 # kind="slash" and entering the branch below. 

375 if wire.author.is_bot or wire.author.is_webhook: 

376 # Webhook traffic is the documented motivation 

377 # (recognized agent personas chatting without an 

378 # @-mention); keep that path at DEBUG since it can 

379 # be high-volume. An unexpected bot author — flagged 

380 # as is_bot but NOT a recognized agent webhook — 

381 # might be a third-party Discord bot or a regression 

382 # mis-classifying a real human; INFO so it shows up 

383 # at production baseline log levels. 

384 is_unrecognized_bot = ( 

385 wire.author.is_bot 

386 and not wire.author.is_webhook 

387 and wire.author.agent_id is None 

388 ) 

389 level = logging.INFO if is_unrecognized_bot else logging.DEBUG 

390 logger.log( 

391 level, 

392 "skipping ambient publish for non-human author " 

393 "event_id=%s author=%s is_bot=%s is_webhook=%s " 

394 "(un-addressed agent chatter does not route)", 

395 wire.event_id, 

396 wire.author.display_name, 

397 wire.author.is_bot, 

398 wire.author.is_webhook, 

399 ) 

400 return 

401 # NOTE: we do NOT populate pending_wires on the ambient 

402 # branch. The router's reply lands on the discard topic 

403 # and is never looked up by event_id; the synthesized 

404 # wires fan-out generates each get their own fresh 

405 # event_id (and that's what the outbox correlates on). 

406 # Inserting here would waste an LRU slot per ambient 

407 # message and could evict legitimate slash entries under 

408 # load. 

409 try: 

410 handle = await self._publish_ambient(wire, phonebook) 

411 except AmbientRosterEmptyError: 

412 # ``_publish_ambient`` already logged the 

413 # operator-actionable ERROR identifying the empty 

414 # roster as the cause. Re-raise without the 

415 # ``logger.exception`` stack trace — the empty-roster 

416 # case is a deployment-config rejection, not a 

417 # broker/runtime failure, so the trace would only 

418 # add noise. The gateway catches this specific type 

419 # and surfaces the misconfiguration to the user. 

420 raise 

421 except Exception: 

422 logger.exception( 

423 "ingress ambient publish failed event_id=%s channel=%s", 

424 wire.event_id, 

425 wire.channel_id, 

426 ) 

427 raise 

428 else: # wire.kind == "slash" 

429 model_settings = self._resolve_model_settings(wire) 

430 temp_instructions = self._resolve_temp_instructions(wire, phonebook) 

431 message_history = await self._build_slash_message_history( 

432 wire, prefetched_history 

433 ) 

434 # Load-bearing ordering: ``put`` MUST precede the 

435 # ``invoke_node`` publish. The outbox consumer reads 

436 # ``PendingWires`` keyed on ``correlation_id`` (= wire 

437 # ``event_id``) to recover the original channel/message/ 

438 # author context for its Discord post. A fast assistant 

439 # reply could race the consumer's lookup if we published 

440 # first and then ``put``. This invariant also matters 

441 # for synthesized wires arriving via 

442 # :mod:`bridge.synthesized`: the synthesized-in consumer 

443 # calls ``ingress.handle(wire)`` here, and the same 

444 # ``put``-before-publish order is what keeps the 

445 # synthesized wire findable by the outbox the moment the 

446 # synthesized assistant replies. 

447 # Snapshot the per-invocation context into the 

448 # PendingEntry so the outbox can rebuild a faithful retry 

449 # envelope on Discord-post failure. All snapshot fields 

450 # mirror what we're about to pass to ``invoke_node``: 

451 # the projected ``message_history`` (frozen as a tuple), 

452 # the peer-roster ``temp_instructions``, and the per-call 

453 # ``model_settings`` (provider-specific thinking-effort). 

454 # Without snapshotting ``model_settings``, a retry would 

455 # silently drop the operator-configured effort tier and 

456 # run at the model client's bake-in default. 

457 self._pending_wires.put( 

458 wire.event_id, 

459 PendingEntry( 

460 wire=wire, 

461 message_history=tuple(message_history), 

462 initial_message_history_length=len(message_history), 

463 temp_instructions=temp_instructions, 

464 model_settings=model_settings, 

465 ), 

466 ) 

467 try: 

468 handle = await self._client.invoke_node( 

469 user_prompt=wire.content, 

470 topic=self._ingress_topic_template.format(cid=wire.channel_id), 

471 correlation_id=wire.event_id, 

472 deps={ 

473 "discord": wire.model_dump(mode="json"), 

474 "phonebook": phonebook_to_deps(phonebook), 

475 **self._memory_prompt_deps(), 

476 }, 

477 output_type=str, 

478 model_settings=model_settings, 

479 temp_instructions=temp_instructions, 

480 message_history=message_history, 

481 ) 

482 except Exception: 

483 # Publish failed; the agent will not run, so no reply 

484 # will ever look up this wire. Free the slot. 

485 self._pending_wires.pop(wire.event_id) 

486 logger.exception( 

487 "ingress slash publish failed event_id=%s channel=%s slash_target=%s", 

488 wire.event_id, 

489 wire.channel_id, 

490 wire.slash_target, 

491 ) 

492 raise 

493 

494 handle._future.cancel()

496 async def _publish_ambient( 

497 self, 

498 wire: WireMessage, 

499 phonebook: list[PhonebookEntry], 

500 ) -> InvocationHandle[Any]: 

501 """Publish an ambient wire to the router's ingress topic. 

502 

503 The original wire, the publisher's phonebook snapshot, and the 

504 channel-history slice all ride on a single ``deps`` dict. The 

505 router run carries those deps forward, so the fan-out consumer 

506 reads them back from ``result.deps`` (calfkit's ``ConsumerContext`` 

507 carries the inbound producer deps — the same dict a tool reads as 

508 ``ctx.deps``). The wire under 

509 ``deps["discord"]`` also mirrors how every other publish path 

510 carries the wire — the synthesized-assistant chain that follows 

511 re-enters :meth:`handle` with ``kind="slash"`` and reads the 

512 wire from deps in the usual place. 

513 

514 Operator-side health signal: we log INFO at every ambient 

515 publish. The synthesized-in consumer (in 

516 :mod:`bridge.synthesized`) logs INFO on every arrival, so 

517 correlating those streams reveals a silent router. Per-reply 

518 WARN tracking is deferred for v1 (see module docstring). 

519 """ 

520 wire_dict = wire.model_dump(mode="json") 

521 phonebook_dict = phonebook_to_deps(phonebook) 

522 temp_instructions = build_router_temp_instructions(phonebook) 

523 # Fetch channel history ONCE here for the entire fan-out. 

524 # 

525 # Eager-fetch-at-ambient-publish-time is intentional even though 

526 # most ambient messages route to zero or one agent (in which 

527 # case we burn ~200ms on a fetch that produces a single 

528 # consumer). The tradeoff buys us: 

529 # 1. Router context — the router LLM sees the recent 

530 # conversation when making its routing decision, which 

531 # improves quality on context-dependent messages 

532 # ("and now do that for next week"). 

533 # 2. Snapshot consistency — every fan-out target sees the 

534 # same history, not slightly-different snapshots that 

535 # each refetch would produce. 

536 # 3. Single REST call regardless of fan-out width — the 

537 # synthesized-in consumer hands the same deps["history"] 

538 # records to each chosen agent, so a fan-out to N agents 

539 # costs one fetch, not N. 

540 # The alternative (lazy fetch at synth-in re-entry) loses all 

541 # three, and the wasted-fetch cost on silent-route decisions is 

542 # small enough that the consistency wins. 

543 records = await self._fetch_ambient_history(wire) 

544 if temp_instructions is None: 

545 # An empty roster is a deployment misconfiguration (no 

546 # assistants registered). Publishing anyway would burn
