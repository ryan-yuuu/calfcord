1"""Channel-history fetching and canonical author-stamped history building. 

2 

3The bridge fetches recent Discord history and builds **one** canonical, 

4author-stamped ``message_history`` per invocation; calfkit's agent-POV 

5projection (``nodes/_projection.py``) re-roles it per viewer at read time 

6(C8/R-A3). Per-author re-roling no longer happens here. Public surface: 

7 

8* :class:`HistoryRecord` — snapshot of one Discord message, with ``is_agent`` 

9 set when the message came from one of the bridge's own persona webhooks 

10 (resolved by ``webhook_id`` against the persona sender's id set, not by 

11 name and not via a registry — R-A3). 

12* :class:`ChannelHistoryFetcher` — thin wrapper around 

13 :meth:`discord.abc.Messageable.history` with a per-channel TTL cache and 

14 graceful Discord-error degradation. Lives in the bridge process. 

15* :func:`build_message_history` — pure function turning a 

16 :class:`HistoryRecord` list into a name-stamped ``list[ModelMessage]`` 

17 (agent turns → :class:`ModelResponse` with ``name=``; humans → 

18 :class:`UserPromptPart` with ``name=``), splicing persisted tool-call 

19 replay deltas before an agent's final-text response. calfkit strips the 

20 ``name`` before any provider sees it. 

21* :class:`DiscordHistoryProvider` — wires the fetcher + transcript store + 

22 builder into the ``MentionRequest`` → ``list[ModelMessage]`` seam the 

23 bridge's ``MentionHandler`` calls. 

24 

25**Why building deliberately does NOT merge adjacent same-role** 

26messages: pydantic-ai's :func:`_clean_message_history` 

27(``calfkit/_vendor/pydantic_ai/_agent_graph.py:1386``) auto-merges 

28adjacent same-type messages with compatible instructions before the 

29provider mapper sees the list. Verified at two call sites: the 

30``UserPromptNode`` (line 213) and ``ModelRequestNode._prepare_request`` 

31(line 526) — both run before any ``model.request()`` call, on every 

32provider. Our constructed messages have ``instructions=None`` and no 

33provider metadata, so the merge conditions are always met. Doing our 

34own merge would be duplicate work. 

35 

36**Why projection DOES drop leading** 

37:class:`~calfkit._vendor.pydantic_ai.messages.ModelResponse` entries: 

38pydantic-ai's ``_clean_message_history`` merges but never drops. If 

39the oldest fetched record is the agent's own webhook reply, the 

40projected history would start with a ``ModelResponse``, and Anthropic 

41rejects request bodies whose first message is ``assistant`` role. Drop 

42is iterative — we keep popping leading responses until we either hit 

43a request or the list is empty (the latter is fine; empty 

44``message_history`` is valid). 

45 

46**Why projection DOES drop empty-content** records: Discord lets 

47through messages with empty ``content`` (system messages like 

48"ryan pinned a message", attachment-only posts). Pydantic-ai's 

49Anthropic mapper has a ``len(user_content_params) > 0`` guard at 

50``models/anthropic.py:740`` that silently skips empty user content, 

51and OpenAI accepts but burns tokens. Dropping at projection time is 

52cleaner and saves Kafka envelope bytes. 

53 

54**Why the fetcher uses ``source_channel_id``** (not the wire's 

55``channel_id``): the bridge's normalizer flattens Discord threads to 

56their parent channel ID for *topic routing* (so all messages in a 

57thread group share one Kafka topic with the parent). For *history 

58fetching* we want the actual channel the message landed in — the 

59thread itself, not the parent. The wire's ``source_channel_id`` field 

60preserves that. 

61 

62**Why the fetcher recovers a thread's starter message**: a Discord 

63thread created from a message ("Start Thread from Message" — how 

64``/task`` threads are made) keeps that starter message in the *parent* 

65channel, not the thread, and its id equals the thread id. A 

66thread-scoped :meth:`~discord.abc.Messageable.history` therefore never 

67returns it, so an agent working a ``/task`` thread would lose the 

68original task statement on every turn after the first (the task text 

69reaches the agent only as the first turn's ``user_prompt``). 

70:meth:`ChannelHistoryFetcher._thread_starter_message` recovers it 

71(in-memory cache first, then one REST fetch from the parent) and 

72:meth:`ChannelHistoryFetcher._do_fetch` prepends it as the oldest 

73record — gated so it appears only on follow-up turns (never duplicating 

74the first-turn trigger) and only when the thread's own history did not 

75already include it (forum threads keep their starter in-thread). 

76"""

109_DEFAULT_CACHE_TTL_SECONDS = 2.0 

110"""Default TTL for the per-channel fetch cache. Tuned to coalesce bursts of 

111fetches that hit :meth:`ChannelHistoryFetcher.fetch` with the same 

112``(source_channel_id, before_message_id, limit)`` key within a few hundred 

113milliseconds (e.g. overlapping turns in the same channel). 2 seconds is 

114long enough to coalesce the burst without serving meaningfully stale 

115data on the next user message.""" 

116 

117_DEFAULT_CACHE_MAX_ENTRIES = 100 

118"""Soft upper bound on cache size. The fetcher evicts LRU entries beyond 

119this. 100 channels-in-flight is far above any realistic concurrent 

120burst on a single bridge process — the bound is a safety net against 

121unbounded growth under pathological load, not a tuned operating point.""" 

122 

123_FORBIDDEN_LOG_DEDUP_MAX = 4096 

124"""Soft upper bound on the Forbidden-error log-dedup set. 

125 

126Sized for channels, not for messages: ``_seen_message_ids`` at 

127:attr:`gateway.DiscordIngressGateway._seen_message_ids` is bounded 

128at :data:`gateway._SEEN_MESSAGE_IDS_CAPACITY` (1024) because message 

129ids churn naturally — old entries are inherently stale. Forbidden 

130state is *sticky*: a channel that lacks Read Message History today 

131lacks it tomorrow, so each channel's entry should persist long enough 

132to keep deduping. With a per-guild ceiling of ~500 channels (Discord 

133limit) plus threads, 4096 channels per bridge process covers any 

134realistic multi-guild deployment without unbounded growth.""" 

135 

136_DISCORD_HISTORY_MAX_LIMIT = 100 

137"""Discord's per-call REST cap for ``channel.history(limit=...)``. The 

138fetcher enforces this so a caller asking for more (e.g. via an 

139operator-edited frontmatter) doesn't surprise-fail at the Discord 

140layer — we cap silently and return up to 100 records."""

157def is_clear_marker(msg: Any, bot_user_id: int | None) -> bool: 

158 """Return ``True`` iff ``msg`` is a bot-posted ``/clear`` boundary marker. 

159

161 predicate (:meth:`DiscordIngressGateway._on_message`): the message must be the 

162 bot's own **non-webhook** post (``webhook_id is None`` and 

163 ``author.id == bot_user_id``) AND its content must exactly equal 

164 :data:`CLEAR_MARKER_TEXT`. Authorship is the load-bearing half — a 

165 regular user cannot post under the bot's user id, and agent personas 

166 post via webhooks (a distinct author id), so the marker cannot be 

167 spoofed by a user typing the sentinel text.

173 non-marker messages short-circuit before any attribute access on 

174 ``author``.

177 reused by other history readers (e.g. the A2A thread reader) without 

178 pulling in fetcher state. 

179 """ 

180 return ( 

181 bot_user_id is not None 

182 and msg.content == CLEAR_MARKER_TEXT 

183 and getattr(msg, "webhook_id", None) is None 

184 and msg.author.id == bot_user_id 

185 )

224class ChannelHistoryFetcher: 

225 """Fetches recent Discord channel history as :class:`HistoryRecord` lists. 

226 

227 Constructed in the bridge process from the gateway's 

228 :class:`discord.Client`. Other deployments do not instantiate this 

229 class — they receive history through Kafka envelopes. 

230 

231 Behavior contract: 

232 

233 * **Bounded**: caller-supplied ``limit`` is clamped to 

234 :data:`_DISCORD_HISTORY_MAX_LIMIT` (Discord's per-call cap). A 

235 ``limit`` of 0 short-circuits (no Discord call) and returns ``[]``. 

236 * **Cached**: results are cached for 

237 :data:`_DEFAULT_CACHE_TTL_SECONDS` keyed on 

238 ``(source_channel_id, before_message_id, limit)``. The cache is a 

239 bounded LRU (:data:`_DEFAULT_CACHE_MAX_ENTRIES`); older entries 

240 evict on insert. Defensive copies on read keep callers from 

241 mutating the cache. 

242 * **Fail-safe**: every Discord error 

243 (:class:`discord.NotFound`, :class:`discord.Forbidden`, 

244 :class:`discord.HTTPException`) is caught and turns into an empty 

245 list with a WARN log. The fetcher never raises into the agent 

246 invocation path — missing history is acceptable; a broken 

247 invocation isn't. 

248 * **Permission dedup**: :class:`discord.Forbidden` (missing Read 

249 Message History) logs once per channel id via a bounded LRU 

250 (:data:`_FORBIDDEN_LOG_DEDUP_MAX`), so misconfigured channels 

251 don't flood logs. 

252 

253 Order returned: oldest-first. Discord's 

254 :meth:`~discord.abc.Messageable.history` returns newest-first by 

255 default; this class reverses internally so the projected 

256 ``message_history`` reads in chronological order (which is what 

257 LLM providers and humans both expect). 

258 """

260 def __init__( 

261 self,

269 # Predicate ``webhook_id -> is one of the bridge's persona webhooks`` 

270 # (R-A3). Sourced from the persona sender's id set (which grows as it 

271 # discovers/creates per-channel webhooks), so it is liveness-independent 

272 # and replaces the deleted registry ``by_display_name`` lookup. 

273 self._owns_webhook = owns_webhook 

274 self._cache_ttl = cache_ttl_seconds 

275 self._cache_max = cache_max_entries

277 tuple[int, int, int], tuple[float, list[HistoryRecord]]

283 # coalesces *sequential* bursts (first caller's result is cached 

284 # by the time the second arrives) — but a simultaneous burst of 

285 # same-key fetches would each race past the empty cache and hit 

286 # Discord independently. The single-flight map ensures one fetch 

287 # serves all N callers.

292 async def fetch( 

293 self, 

294 *, 

295 source_channel_id: int, 

296 before_message_id: int, 

297 limit: int, 

298 bypass_cache: bool = False, 

299 ) -> list[HistoryRecord]: 

300 """Fetch up to ``limit`` records older than ``before_message_id``. 

301 

302 Args: 

303 source_channel_id: The actual Discord channel the triggering 

304 message landed in (thread or top-level). The bridge's 

305 normalizer collapses threads to their parent for 

306 topic-routing purposes; this argument must be the 

307 un-collapsed value so the right history is fetched. 

308 before_message_id: Discord snowflake to anchor the fetch. 

309 The returned records are strictly older than this id 

310 (Discord's ``before=`` is exclusive). Typical caller 

311 passes ``wire.message_id``. 

312 limit: Maximum records to return. Clamped to 

313 :data:`_DISCORD_HISTORY_MAX_LIMIT`. ``0`` is treated as 

314 "history disabled" and short-circuits without a Discord 

315 call. Exception: for a message-started thread the recovered 

316 starter message is prepended *after* the ``limit``-bounded 

317 fetch, so the returned list can hold ``limit + 1`` records 

318 (the starter plus ``limit`` in-thread messages). Callers 

319 that need a hard cap re-trim with ``records[-N:]`` — which 

320 drops the oldest, i.e. the starter, first (see 

321 :meth:`_do_fetch` and the prepend comment there). 

322 bypass_cache: When ``True`` (default ``False``), skip the 

323 read-side LRU lookup AND the write-back. The fetch 

324 always goes to Discord (subject to single-flight) and 

325 its result is never stored in the LRU. Single-flight 

326 registration still happens, so concurrent bypass 

327 callers for the same key share one Discord call — 

328 preserving the fan-out coalescing invariant. Use this 

329 for sources whose freshness matters more than their 

330 LRU hit-rate (e.g. A2A thread reads, where the caller's 

331 own request was just posted and the LRU's 2-second 

332 window would serve stale records). 

333 

334 Returns: 

335 Oldest-first list of :class:`HistoryRecord`. Empty on any 

336 failure or cap. The contract is "never raises into the 

337 invocation path" — any exception from Discord or from 

338 record construction is logged and absorbed into an empty 

339 list. 

340 """ 

341 limit = min(max(0, limit), _DISCORD_HISTORY_MAX_LIMIT) 

342 if limit == 0: 

343 return [] 

344 

345 key = (source_channel_id, before_message_id, limit) 

346 

347 # Cache check (fresh) — first try. Skipped on bypass. 

348 if not bypass_cache: 

349 cached = self._cache.get(key) 

350 if cached is not None and monotonic() - cached[0] < self._cache_ttl: 

351 self._cache.move_to_end(key) 

352 return list(cached[1]) 

353 

354 # Single-flight check — if another coroutine is already fetching 

355 # this same key, join its future instead of starting a parallel 

356 # Discord call. This is the core of the fan-out coalescing 

357 # guarantee documented at :attr:`_in_flight`. Bypass callers 

358 # still join in-flight fetches so concurrent A2A reads on the 

359 # same thread share one Discord round-trip; the resulting list 

360 # is fresh-from-Discord either way. 

361 in_flight = self._in_flight.get(key) 

362 if in_flight is not None: 

363 return list(await in_flight) 

364 

365 loop = asyncio.get_running_loop() 

366 future: asyncio.Future[list[HistoryRecord]] = loop.create_future() 

367 self._in_flight[key] = future 

368 try: 

369 records = await self._do_fetch( 

370 source_channel_id=source_channel_id, 

371 before_message_id=before_message_id, 

372 limit=limit, 

373 )

375 # future for any concurrent followers, return a defensive 

376 # copy to the leader. These steps live INSIDE the try block 

377 # so that any exception they raise (cache helper bug, 

378 # ``set_result`` on a misused future, etc.) flows through 

379 # the same contract-enforcing except handlers below — the 

380 # "fetcher never raises into invocation path" guarantee 

381 # covers the full body, not just the Discord call. 

382 if not bypass_cache: 

383 self._cache_and_return(key, monotonic(), records) 

384 if not future.done(): 384 ↛ 386line 384 didn't jump to line 386 because the condition on line 384 was always true

385 future.set_result(records) 

386 return list(records) 

387 except asyncio.CancelledError: 

388 # The leader's task is being cancelled (e.g., bridge 

389 # shutdown). Do NOT propagate this cancellation to passive 

390 # followers awaiting the shared future — they have their 

391 # own cancellation scopes and would otherwise see a 

392 # ``CancelledError`` they never asked for. Resolve the 

393 # shared future with the documented empty-history sentinel 

394 # so followers complete normally; re-raise so the leader's 

395 # own stack sees the cancel. 

396 if not future.done(): 396 ↛ 398line 396 didn't jump to line 398 because the condition on line 396 was always true

397 future.set_result([]) 

398 raise 

399 except Exception: 

400 # Defense-in-depth: :meth:`_do_fetch` is contractually 

401 # total — its own try/except absorbs every documented 

402 # Discord error class into ``[]``. Reaching this branch 

403 # means a truly unexpected exception (a bug in 

404 # :meth:`_to_record`, a future ``discord.py`` shape change 

405 # that slipped past the inner sweep, or a defect in the 

406 # cache helper). Log loudly at this layer so the failure is 

407 # observable, then degrade per the public fetcher contract 

408 # ("never raises into invocation path"): every caller — 

409 # the leader AND any concurrent followers — gets ``[]``. 

410 logger.exception( 

411 "history fetch single-flight raised unexpectedly; " 

412 "serving [] to all callers source_channel_id=%d " 

413 "before_message_id=%d limit=%d", 

414 source_channel_id, 

415 before_message_id, 

416 limit, 

417 ) 

418 if not future.done(): 418 ↛ 420line 418 didn't jump to line 420 because the condition on line 418 was always true

419 future.set_result([]) 

420 return [] 

421 finally: 

422 self._in_flight.pop(key, None)

424 async def _do_fetch( 

425 self, 

426 *, 

427 source_channel_id: int, 

428 before_message_id: int, 

429 limit: int, 

430 ) -> list[HistoryRecord]: 

431 """Inner fetch implementation — never invoked concurrently for 

432 the same key (the outer :meth:`fetch` enforces single-flight). 

433 

434 Catches every documented Discord error class plus a defensive 

435 ``Exception`` sweep around record construction so the public 

436 ``fetch`` contract ("never raises into invocation path") 

437 holds even if a future ``discord.py`` release returns 

438 unexpected ``Message`` shapes. 

439 

440 Records are also truncated at the most recent ``/clear`` marker 

441 (see :func:`is_clear_marker` and the inline comment at the scan 

442 site for the rationale). 

443 

444 For a message-started thread, the starter message — which lives in 

445 the parent channel and is therefore absent from 

446 ``thread.history()`` — is recovered via 

447 :meth:`_thread_starter_message` and prepended as the oldest record 

448 when it falls within this fetch's exclusive ``before=`` window and 

449 the thread's own history did not already include it. See the inline 

450 comment at the prepend site. 

451 """ 

452 channel = await self._resolve_channel(source_channel_id) 

453 if channel is None: 

454 return [] 

455 

456 try: 

457 messages = [ 

458 m 

459 async for m in channel.history( 

460 limit=limit, 

461 before=discord.Object(id=before_message_id), 

462 ) 

463 ] 

464 except discord.Forbidden: 

465 self._log_forbidden_once(source_channel_id) 

466 return [] 

467 except discord.NotFound: 

468 logger.warning( 

469 "channel_id=%d: history fetch returned 404; channel may have been deleted", 

470 source_channel_id, 

471 ) 

472 return [] 

473 except discord.HTTPException as e: 

474 logger.warning( 

475 "channel_id=%d: history fetch failed status=%s: %s", 

476 source_channel_id, 

477 e.status, 

478 e, 

479 ) 

480 return [] 

481 

482 # discord.py returns newest-first; reverse for chronological order. 

483 # Then drop everything up to and including the most recent ``/clear`` 

484 # marker (see :func:`is_clear_marker`) so a prior operator ``/clear`` 

485 # bounds the history every downstream consumer sees — the marker 

486 # itself is excluded too. Scanning the raw ``discord.Message`` list 

487 # (rather than carrying a flag on :class:`HistoryRecord`) keeps the 

488 # marker concept local to the fetcher and the record schema 

489 # unchanged; the bot-author identity bits the recognizer needs 

490 # (``author.id`` / ``webhook_id``) live only on the message, not the 

491 # projected record. The scan + projection share one defensive 

492 # ``Exception`` sweep: if a future ``discord.py`` returns a 

493 # ``Message`` shape missing an attribute we read, we degrade to 

494 # empty history rather than raise into the invocation path 

495 # (the "never raises" contract). 

496 bot_user = self._client.user 

497 bot_user_id = bot_user.id if bot_user is not None else None 

498 try: 

499 ordered = list(reversed(messages)) 

500 # Recover a message-started thread's starter message and prepend 

501 # it as the oldest entry. The starter lives in the PARENT channel 

502 # (see :meth:`_thread_starter_message`), so ``thread.history()`` 

503 # never returns it — without this, agents in a ``/task`` thread 

504 # lose the original task statement on every turn after the first. 

505 # 

506 # Two gates keep it exactly-once and in the right place: 

507 # * ``starter.id < before_message_id`` mirrors Discord's 

508 # exclusive ``before=`` rule applied to the anchor. On the 

509 # first ``/task`` turn the anchor IS the triggering message 

510 # (``before_message_id == thread.id == starter.id``) and is 

511 # supplied as the ``user_prompt``, so it stays excluded just 

512 # as a normal trigger would; on later turns it is included. 

513 # * the membership check skips the prepend when the fetched 

514 # history already contains the starter — forum-post threads 

515 # keep their starter in-thread (id == thread.id), so 

516 # ``thread.history()`` returns it and prepending would 

517 # duplicate it. 

518 # Prepended BEFORE the ``/clear`` scan, so a ``/clear`` inside the 

519 # thread truncates the task statement too ("clear means clear"). 

520 # Inside this defensive try so a malformed starter shape degrades 

521 # to empty history rather than raising into the invocation path. 

522 starter = await self._thread_starter_message(channel) 

523 if ( 

524 starter is not None 

525 and starter.id < before_message_id 

526 and not any(m.id == starter.id for m in ordered) 

527 ): 

528 ordered.insert(0, starter) 

529 cut = -1 

530 for i in range(len(ordered) - 1, -1, -1): 

531 if is_clear_marker(ordered[i], bot_user_id): 

532 cut = i 

533 break 

534 return [self._to_record(m) for m in ordered[cut + 1:]] 

535 except Exception: 

536 logger.exception( 

537 "channel_id=%d: failed to project messages into HistoryRecords; " 

538 "returning empty history. This indicates an unexpected " 

539 "discord.Message shape; investigate.",

544 async def _thread_starter_message(self, channel: Any) -> discord.Message | None: 

545 """Recover a message-started thread's starter message, or ``None``. 

546 

547 A thread created from a message (Discord's "Start Thread from 

548 Message" — how ``/task`` threads and manually message-started 

549 threads are made) keeps that starter message in the **parent 

550 channel**, not the thread, so :meth:`discord.Thread.history` never 

551 yields it. The starter's id equals the thread id (per discord.py's 

552 :attr:`discord.Thread.starter_message`: "the thread starter message 

553 ID is the same ID as the thread"). 

554 

555 Recovery order: 

556 

557 1. ``channel.starter_message`` — a read of discord.py's 

558 **client-wide** in-memory message cache (the single 

559 ``max_messages``-bounded deque in ``ConnectionState``, default 

560 1000, shared across every channel and populated from 

561 ``MESSAGE_CREATE`` gateway events). Opportunistic: on a busy bot 

562 the global buffer evicts the starter quickly, so this is usually 

563 a miss on follow-up turns (and always after a bridge restart), in 

564 which case we fall through to the REST path. 

565 2. ``parent.fetch_message(channel.id)`` — one REST call, the 

566 realistic common path (the starter id equals the thread id). 

567 

568 Duck-typed on ``parent_id`` (mirrors 

569 :func:`~calfcord.bridge.normalizer._resolve_channel_id`, 

570 the codebase's thread-detection primitive) so a non-thread channel 

571 returns ``None`` without a REST call and tests can use plain 

572 ``SimpleNamespace`` fakes. 

573 

574 Total — every failure degrades to ``None`` (today's thread-only 

575 history), logged like the sibling fetch failures in 

576 :meth:`_do_fetch`: 

577 

578 * non-thread channel (no ``parent_id``) → ``None``, no REST; 

579 * a cached starter → returned directly (no REST); 

580 * a forum-channel parent (no ``fetch_message``) → ``None`` (forum 

581 threads keep their starter in-thread, so no recovery is needed); 

582 * thread with no fetchable starter — created standalone, or the 

583 starter was deleted (``discord.NotFound``) → ``None``; 

584 * uncached parent that cannot be resolved → ``None``; 

585 * ``discord.Forbidden`` → ``None``, deduped once per parent id; 

586 * ``discord.HTTPException`` → ``None``, WARN. 

587 """ 

588 parent_id = getattr(channel, "parent_id", None) 

589 if parent_id is None: 

590 return None # not a thread 

591 # discord.py's client-wide in-memory message cache — see docstring. 

592 # Opportunistic — usually a miss on later turns / after a restart, in 

593 # which case we fall to the REST fetch below. 

594 starter = getattr(channel, "starter_message", None) 

595 if starter is not None: 

596 return starter # in-memory cache hit — no REST 

597 parent = getattr(channel, "parent", None) 

598 if parent is None: 

599 parent = await self._resolve_channel(parent_id) 

600 # A forum-channel parent is not messageable and has no 

601 # ``fetch_message`` — guard so we degrade to ``None`` instead of 

602 # raising (forum threads keep their starter in-thread anyway). 

603 if parent is None or not hasattr(parent, "fetch_message"): 

604 return None 

605 try: 

606 return await parent.fetch_message(channel.id) 

607 except discord.NotFound: 

608 # Standalone thread (not created from a message) or the starter 

609 # was deleted — not an error; there is simply no anchor to add. 

610 # DEBUG (not WARN) so the common "this thread has no message 

611 # starter" case never spams logs, while still leaving a trail an 

612 # operator can switch on to distinguish it from the rare 

613 # "starter was deleted" case when chasing "the agent forgot the 

614 # task statement". 

615 logger.debug( 

616 "channel_id=%d: no fetchable thread starter (standalone " 

617 "thread or starter deleted); skipping anchor prepend", 

618 channel.id, 

619 ) 

620 return None 

621 except discord.Forbidden: 

622 self._log_forbidden_once(parent_id) 

623 return None 

624 except discord.HTTPException as e: 

625 logger.warning( 

626 "channel_id=%d: thread starter-message fetch failed status=%s: %s", 

627 channel.id, 

628 e.status, 

629 e, 

630 ) 

631 return None

731def build_message_history( 

732 records: Sequence[HistoryRecord], 

733 *, 

734 hydration: Mapping[int, list[ModelMessage]] | None = None, 

735) -> list[ModelMessage]: 

736 """Build ONE canonical, author-stamped ``message_history`` from ``records``. 

737 

738 POV-agnostic (C8/R-A3): every agent turn becomes a :class:`ModelResponse` 

739 stamped ``name=<agent name>`` and every human turn a :class:`ModelRequest` 

740 whose :class:`UserPromptPart` is stamped ``name=<display name>``. calfkit's 

741 agent-POV projection (``nodes/_projection.py``) re-roles this per viewer at 

742 read time — the running agent's own turns stay responses, every other author 

743 is surfaced as an attributed user turn — and strips the ``name`` before any 

744 provider sees it. So the bridge builds the history once, independent of which 

745 agent it is sent to (no app-side per-viewer projection). 

746 

747 Args: 

748 records: Oldest-first :class:`HistoryRecord` list from 

749 :meth:`ChannelHistoryFetcher.fetch`. 

750 hydration: Optional tool-call replay map, ``message_id -> the stored, 

751 name-stamped structured delta`` for that agent reply. When an agent 

752 record's ``message_id`` is in the map, the delta is spliced in 

753 IMMEDIATELY BEFORE that record's final-text :class:`ModelResponse`, 

754 so the agent re-sees the tool calls/returns it made on that turn. The 

755 delta's responses must already carry the agent's ``name`` (see 

756 :meth:`DiscordHistoryProvider._build_replay_hydration`) so calfkit 

757 attributes them to the right viewer. ``None`` (default) reproduces 

758 the no-replay output exactly. Pure: never touches the DB. 

759 

760 Returns: 

761 A list suitable for ``start(message_history=...)``. May be empty. 

762 

763 Deliberately does NOT merge consecutive same-role entries — pydantic-ai's 

764 ``_clean_message_history`` (``_agent_graph.py:1386``) does that before any 

765 provider mapper runs (our messages satisfy its merge conditions). It DOES 

766 drop empty/whitespace-only content (Anthropic skips zero-block user messages; 

767 OpenAI wastes tokens) and DOES drop leading :class:`ModelResponse` entries 

768 iteratively — calfkit's projection keeps a self-authored leading response as 

769 a response, and Anthropic rejects a history whose first message is 

770 ``assistant``. 

771 """ 

772 out: list[ModelMessage] = [] 

773 seen_request = False 

774 for r in records: 

775 if not r.content.strip(): 

776 continue 

777 if r.is_agent: 

778 # Tool-call replay: splice this agent reply's persisted delta just 

779 # BEFORE its final-text response so the agent re-sees the tool 

780 # calls/returns it made. Gate on ``seen_request``: an agent reply 

781 # with no preceding user ``ModelRequest`` is a *leading* response 

782 # the trailing drop below removes — splicing its delta there would 

783 # orphan a tool-return ``ModelRequest`` (a ``tool_result`` with no 

784 # matching ``tool_use``) at the head → provider 400. 

785 if hydration is not None and seen_request: 

786 replay = hydration.get(r.message_id) 

787 if replay: 

788 out.extend(replay) 

789 out.append( 

790 ModelResponse(parts=[TextPart(content=r.content)], name=r.author_display_name) 

791 ) 

792 else: 

793 out.append( 

794 ModelRequest(parts=[UserPromptPart(content=r.content, name=r.author_display_name)]) 

795 ) 

796 seen_request = True 

797 while out and isinstance(out[0], ModelResponse): 

798 out.pop(0) 

799 return out

802REPLAY_TOOL_RETURN_MAX_CHARS: Final[int] = 6000 

803"""Per-tool-return character cap applied to replayed (hydrated) tool returns 

804before they re-enter an agent's ``message_history``. 

805 

806Deliberately distinct from — and much larger than — the 2000-char Discord 

807display budget: this bounds the *LLM context* a replayed turn re-injects, not a 

808*display*. Reusing a display-sized cap would lobotomize the tool context the 

809model reasons over on the next turn (the whole point of replay). Only oversized 

810individual tool returns are trimmed; per R-A5 there is no global history ceiling, 

811and this per-return cap is the one backstop kept (it bounds the realistic 

812envelope blow-up — a giant tool output spliced back in).""" 

813 

814_REPLAY_TRUNCATION_MARKER: Final[str] = "\n…(truncated)" 

815"""Visible marker appended to a tool return truncated to 

816:data:`REPLAY_TOOL_RETURN_MAX_CHARS` so the model can tell the content was cut 

817rather than genuinely short.""" 

818 

819 

820def _truncate_replay_tool_returns(messages: list[ModelMessage]) -> list[ModelMessage]: 

821 """Cap oversized tool-return payloads in a replayed delta. 

822 

823 Walks ``messages`` (the deserialized structured slice of a prior turn) and, 

824 for every :class:`BaseToolReturnPart` whose model-facing string 

825 (``part.model_response_str()``) exceeds :data:`REPLAY_TOOL_RETURN_MAX_CHARS`, 

826 replaces the part with an immutable copy whose ``content`` is the truncated 

827 string plus :data:`_REPLAY_TRUNCATION_MARKER`. Non-mutating: rebuilds only 

828 the parts/messages that need it and returns a NEW list, so callers can splice 

829 it without aliasing the input. 

830 """ 

831 out: list[ModelMessage] = [] 

832 for msg in messages: 

833 new_parts: list[Any] | None = None 

834 parts = msg.parts 

835 for idx, part in enumerate(parts): 

836 if isinstance(part, BaseToolReturnPart): 

837 rendered = part.model_response_str() 

838 if len(rendered) > REPLAY_TOOL_RETURN_MAX_CHARS: 

839 if new_parts is None: 839 ↛ 841line 839 didn't jump to line 841 because the condition on line 839 was always true

840 new_parts = list(parts) 

841 head = REPLAY_TOOL_RETURN_MAX_CHARS - len(_REPLAY_TRUNCATION_MARKER) 

842 truncated = rendered[: max(head, 0)] + _REPLAY_TRUNCATION_MARKER 

843 new_parts[idx] = dataclasses.replace(part, content=truncated) 

844 if new_parts is None: 

845 out.append(msg) 

846 else: 

847 out.append(dataclasses.replace(msg, parts=new_parts)) 

848 return out

853 

854 A persisted delta (dumped from the agent's own run) has no ``name`` on its

869class DiscordHistoryProvider: 

870 """Bridge-side ``MentionRequest`` → ``list[ModelMessage]`` history seam. 

871 

872 Wires :class:`ChannelHistoryFetcher` (Discord fetch + ``/clear`` truncation + 

873 thread-starter recovery), the transcript store (tool-call replay deltas), and 

874 :func:`build_message_history` (canonical name-stamped build). Satisfies the 

875 ``HistoryProvider`` protocol the bridge's ``MentionHandler`` calls. 

876 """ 

877 

878 def __init__( 

879 self, 

880 fetcher: ChannelHistoryFetcher, 

881 transcript_store: TranscriptStoreLike, 

882 *, 

883 limit: int = _DISCORD_HISTORY_MAX_LIMIT, 

884 ) -> None: 

885 self._fetcher = fetcher 

886 self._transcript_store = transcript_store 

887 # Per R-A5 there is no global history ceiling; ``limit`` is just the 

888 # Discord per-call REST cap (≤100). The fetched window is used as-is. 

889 self._limit = limit 

890 

891 async def message_history(self, req: MentionRequest) -> list[ModelMessage]: 

892 records = await self._fetcher.fetch( 

893 source_channel_id=req.source_channel_id, 

894 before_message_id=req.message_id, 

895 limit=self._limit, 

896 ) 

897 hydration = await self._build_replay_hydration(records) 

898 return build_message_history(records, hydration=hydration) 

899 

900 async def _build_replay_hydration( 

901 self, records: Sequence[HistoryRecord] 

902 ) -> dict[int, list[ModelMessage]] | None: 

903 """Join the fetched agent-reply records against the transcript store. 

904 

905 For every agent record (``is_agent``) with a persisted transcript row, 

906 deserialize its structured delta, trim oversized tool returns to 

907 :data:`REPLAY_TOOL_RETURN_MAX_CHARS`, and stamp the agent's ``name`` onto 

908 every :class:`ModelResponse` in the delta so calfkit's POV projection 

909 attributes those tool calls to that agent. Never DB-scans — the read 

910 scope is the surviving (``/clear``-truncated) record set, so a cleared 

911 reply is simply never looked up (``/clear`` correctness for free). 

912 

913 Returns ``None`` (⇒ no replay) when the store is disabled/empty, no agent 

914 record has a row, or every row fails to deserialize. Best-effort: a 

915 single corrupt blob is logged and skipped. 

916 """ 

917 agent_names = {r.message_id: r.author_display_name for r in records if r.is_agent} 

918 if not agent_names: 

919 return None 

920 # The store keys ``final_message_id`` as TEXT (snowflake precision); join 

921 # on the str form and map back to int to match ``HistoryRecord.message_id``. 

922 rows = await self._transcript_store.get_by_final_message_ids( 

923 [str(mid) for mid in agent_names] 

924 ) 

925 if not rows: 

926 return None 

927 hydration: dict[int, list[ModelMessage]] = {} 

928 for final_message_id_str, row in rows.items(): 

929 message_id = int(final_message_id_str) 

930 agent_name = agent_names[message_id] 

931 try: 

932 msgs = list(ModelMessagesTypeAdapter.validate_json(row.delta_json)) 

933 msgs = _truncate_replay_tool_returns(msgs) 

934 msgs = _stamp_response_names(msgs, agent_name) 

935 except Exception: 

936 # A single unparseable/malformed blob must not sink the whole 

937 # invocation: skip just this reply's replay. The row was written 

938 # from a live turn, so this is unexpected — log for investigation. 

939 logger.exception( 

940 "replay hydration skipped reply_id=%s agent=%s: failed to " 

941 "deserialize/trim/stamp stored delta; this turn keeps only " 

942 "its final text in replay", 

943 final_message_id_str, 

944 agent_name, 

945 ) 

946 continue 

947 hydration[message_id] = msgs 

948 return hydration or None
