1"""Shared retry-with-feedback policy for Discord-rejected LLM replies. 

2 

3Two callers post LLM-generated text to Discord and both want the same 

4"if Discord rejects it, tell the LLM and let it adapt" loop: 

5 

6* :mod:`calfcord.bridge.outbox` — channel replies, posted 

7 fire-and-forget via ``client.send`` on retry. 

8* :mod:`calfcord.tools.private_chat` — A2A audit 

9 projections, posted synchronously inside the caller's 

10 ``execute`` RPC on retry. 

11 

12The two orchestration mechanisms cannot share their transport (one is 

13async-via-Kafka-consumer, the other is await-inside-RPC), but they DO 

14share their **policy** and their **content construction**: the retry 

15budget, the error classification, the ``<system-reminder>`` wording, 

16the ``message_history`` shape, and the chunk-split fallback. This 

17module is the single source of truth for those. 

18 

19Process boundary: this module sits beneath both consumers and depends 

20only on stdlib, ``discord``, and ``pydantic_ai`` message types — no 

21project imports — so both ``bridge`` and ``tools`` can import it 

22without inducing a cycle. 

23"""

39# --- Constants ---------------------------------------------------------------- 

40 

41MAX_REPLY_RETRY_ATTEMPTS: Final[int] = 2 

42"""Number of LLM retries triggered after the original post failure. 

43Total LLM attempts before falling back to chunk-splitting = 

441 (original) + MAX_REPLY_RETRY_ATTEMPTS = 3. Picked at 2 because: 

45 

46* Discord 4xx errors (length, formatting) are usually self-correcting 

47 on attempt 2 once the LLM is told the problem. 

48* Each retry is a full LLM round-trip (~5-15s); 2 retries adds at most 

49 ~30s before chunked fallback. 3+ retries makes the user wait too 

50 long with no visible signal. 

51* Bounded LLM cost: pathological retry loops cost ~3x a normal 

52 invocation, never unbounded.""" 

53 

54CHUNK_SAFE_SIZE: Final[int] = 1990 

55"""Max chars per chunk in the chunk-split fallback. Discord's hard 

56content limit is 2000; the 10-char safety buffer absorbs the occasional 

57emoji / encoding surprise that tips a 1999-char string over the limit.""" 

58 

59NON_AGENT_FIXABLE_STATUSES: Final[frozenset[int]] = frozenset({ 

60 401, # unauthorized — bot token invalid 

61 403, # forbidden — missing Manage Webhooks / View Channel / etc. 

62 404, # not found — channel or webhook deleted 

63 429, # rate limited — discord.py already retried internally 

64}) 

65"""Discord HTTP statuses where retrying the agent with a revised reply 

66cannot possibly succeed. These are infrastructure / permission errors 

67that require operator action, not agent content adjustment. Callers 

68log WARN and drop the reply on these statuses. 

69 

705xx is NOT in this set — :func:`classify_error` returns ``"transient"`` 

71for those, because the inner per-send retry that callers already do 

72smooths transient 5xx; only a *persistent* 5xx reaches this layer. 

73 

74:class:`discord.RateLimited` is also classified as ``"drop"`` even 

75though it has no HTTP ``status`` (it inherits from 

76:class:`discord.DiscordException` directly); see :func:`classify_error`.""" 

77 

78_RETRY_REMINDER_OVERRIDES: dict[tuple[int, int], str] = {} 

79"""Per-(HTTP status, JSON error code) overrides for the retry-reminder 

80text. Empty by default — the generic template surfaces Discord's own 

81error message to the LLM, which modern frontier models reliably parse 

82and adapt to. Populate only when empirical evidence shows the LLM 

83needs more pointed guidance for a specific Discord error code. 

84 

85Format: ``(status, code): "Custom reminder body."`` Both status and 

86code are concrete integers — no wildcard support in v1."""

92def classify_error( 

93 error: discord.DiscordException, 

94) -> Literal["drop", "transient", "agent_fixable"]: 

95 """The single source of truth for what to do with a failed Discord post. 

96 

97 Returns: 

98 * ``"drop"`` — non-agent-fixable infra (statuses in 

99 :data:`NON_AGENT_FIXABLE_STATUSES`, :class:`discord.RateLimited`, 

100 or a :class:`discord.DiscordException` that's neither 

101 :class:`HTTPException` nor :class:`RateLimited`). Caller should 

102 log+abandon — the LLM cannot fix permission, auth, or 

103 rate-limit issues by re-thinking its reply. 

104 * ``"transient"`` — 5xx that survived the sender's inner retry. 

105 Caller should log+abandon (bridge) or raise (A2A); a 

106 content-retry won't fix a Discord-side outage. 

107 * ``"agent_fixable"`` — 4xx the LLM can plausibly correct 

108 (content-too-long, invalid embed, banned formatting). Caller 

109 should fire a retry-with-feedback. 

110 """ 

111 if isinstance(error, discord.RateLimited): 

112 return "drop" 

113 if not isinstance(error, discord.HTTPException): 

114 return "drop" 

115 if error.status in NON_AGENT_FIXABLE_STATUSES: 

116 return "drop" 

117 if error.status >= 500: 

118 return "transient" 

119 return "agent_fixable"

122# --- Retry envelope construction ---------------------------------------------

124 

125def build_retry_reminder( 

126 error: discord.HTTPException, 

127 failed_text: str, 

128) -> str: 

129 """Build the system-reminder-tagged user message for an agent retry. 

130 

131 Generic by design: the LLM sees the literal Discord error text in 

132 a ``<system-reminder>`` block and is trusted to adapt. Modern 

133 frontier LLMs reliably parse Discord's own error strings (e.g. 

134 ``"Must be 2000 or fewer in length"``, ``"Cannot send an empty 

135 message"``, ``"Invalid embed URL"``) and adjust their next reply 

136 accordingly. No per-error-code customization is needed in v1; the 

137 override map slot at :data:`_RETRY_REMINDER_OVERRIDES` exists for 

138 future empirical cases where the generic message is insufficient. 

139 

140 The ``<system-reminder>`` tag pattern is a convention frontier 

141 models trained with system-reminder-style data typically treat as 

142 out-of-band metadata even though it occupies a ``user``-role slot 

143 on the wire. The explicit "Do NOT mention this error" instruction 

144 inside the reminder body is the actual enforcement mechanism; 

145 the tag wrapper is the visual cue that helps the model recognize 

146 the convention. 

147 

148 Args: 

149 error: The :class:`discord.HTTPException` raised by the 

150 persona-sender. The status, code, and body text are all 

151 surfaced to the LLM. 

152 failed_text: The exact reply text the agent emitted that 

153 Discord rejected. Used in the reminder to give the LLM 

154 length-context (``"length: 3187 chars"``) without 

155 duplicating the full failed content (which appears 

156 separately in the retry envelope's ``message_history`` 

157 as a ``ModelResponse``). 

158 

159 Returns: 

160 A string suitable to pass as ``user_prompt`` to 

161 :meth:`Client.send` (bridge) or 

162 :meth:`Client.execute` (A2A) for the retry envelope. 

163 """ 

164 override = _RETRY_REMINDER_OVERRIDES.get((error.status, error.code)) 

165 if override is not None: 

166 body = override 

167 else: 

168 # ``discord.HTTPException.text`` is the raw JSON-body text from 

169 # Discord (e.g. ``"Invalid Form Body\nIn content: Must be 2000 

170 # or fewer in length."``). Falls back to ``str(error)`` which 

171 # is discord.py's formatted ``"status: code: text"``. 

172 raw = error.text or str(error) 

173 body = ( 

174 f"Your previous reply (length: {len(failed_text)} chars) was " 

175 f"rejected by Discord. The exact error:\n\n" 

176 f" HTTP {error.status}: {raw}\n\n" 

177 f"Please respond again to the user's original question, " 

178 f"addressing the specific issue above. For example, if the " 

179 f"content was too long, be more concise; if it contained " 

180 f"banned formatting, rephrase without it." 

181 ) 

182 return ( 

183 "<system-reminder>\n" 

184 f"{body}\n\n" 

185 "This reminder is system-level — the user does NOT see it. " 

186 "Do NOT mention this error or that you are retrying.\n" 

187 "</system-reminder>" 

188 )

191def build_retry_history( 

192 *, 

193 original_history: Sequence[ModelMessage], 

194 original_user_prompt: str, 

195 failed_text: str, 

196) -> list[ModelMessage]: 

197 """Build the retry envelope's ``message_history``. 

198 

199 Used identically by both callers: original history, then the user's 

200 original prompt as ``ModelRequest``, then the LLM's rejected 

201 attempt as ``ModelResponse``. The new ``<system-reminder>`` 

202 user prompt is passed separately by the caller (as the retry 

203 invocation's ``user_prompt``) — pydantic-ai's 

204 ``_clean_message_history`` then merges any adjacent same-role 

205 parts before the provider mapper sees the list, so the LLM ends 

206 up with a well-formed alternating conversation that ends on the 

207 system-reminder user turn. 

208 """ 

209 return [ 

210 *original_history, 

211 ModelRequest(parts=[UserPromptPart(content=original_user_prompt)]), 

212 ModelResponse(parts=[TextPart(content=failed_text)]), 

213 ]

219def chunk_split(text: str, *, max_chars: int = CHUNK_SAFE_SIZE) -> list[str]: 

220 """Split ``text`` into pieces each ≤ ``max_chars`` for posting as 

221 consecutive Discord messages. 

222 

223 Boundary search is greedy from the largest unit down: paragraph 

224 (``"\\n\\n"``) → line (``"\\n"``) → sentence (``". "``) → word 

225 (``" "``) → hard cut. The search refuses to split earlier than 

226 ``max_chars // 2`` so we don't produce a tiny first chunk 

227 followed by a huge tail. 

228 

229 Each chunk is right-stripped of trailing whitespace. The split 

230 preserves all non-boundary characters — joining chunks back with 

231 the boundary that produced each cut reconstructs (modulo 

232 boundary whitespace) the original text. 

233 

234 Args: 

235 text: The full text to split. May be empty (returns ``[]``). 

236 max_chars: Maximum characters per chunk. Defaults to 

237 :data:`CHUNK_SAFE_SIZE` (1990) — Discord's 2000-char 

238 limit with a 10-char safety buffer. 

239 

240 Returns: 

241 A list of chunks in original order. If ``text`` already fits, 

242 returns ``[text]``. An empty string returns ``[]``. 

243 """ 

244 if not text: 

245 return [] 

246 if len(text) <= max_chars: 

247 return [text] 

248 

249 chunks: list[str] = [] 

250 remaining = text 

251 min_split = max(1, max_chars // 2) 

252 

253 while remaining: 253 ↛ 278line 253 didn't jump to line 278 because the condition on line 253 was always true

254 if len(remaining) <= max_chars: 

255 stripped = remaining.rstrip() 

256 if stripped: 256 ↛ 258line 256 didn't jump to line 258 because the condition on line 256 was always true

257 chunks.append(stripped) 

258 break 

259 

260 candidate = remaining[:max_chars] 

261 cut_at = -1 

262 # Prefer larger structural boundaries. 

263 for separator in ("\n\n", "\n", ". ", " "): 

264 idx = candidate.rfind(separator) 

265 if idx >= min_split: 

266 cut_at = idx + len(separator) 

267 break 

268 

269 if cut_at < 0: 

270 # No good boundary found; hard cut at max_chars. 

271 cut_at = max_chars 

272 

273 chunk = remaining[:cut_at].rstrip() 

274 if chunk: 274 ↛ 276line 274 didn't jump to line 276 because the condition on line 274 was always true

275 chunks.append(chunk) 

276 remaining = remaining[cut_at:] 

277 

278 return chunks
